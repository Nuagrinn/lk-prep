# Кодирование и эволюция

Четвертая глава DDIA про то, как данные переживают изменения программы. Система
не обновляется одним мгновенным щелчком: старый код и новый код часто работают
одновременно, старые данные лежат в базе, новые клиенты общаются со старыми
серверами, а сообщения в очереди могут быть записаны одной версией и прочитаны
другой.

Главная мысль главы:

> Если приложение должно развиваться без остановки мира, форматы данных и
> протоколы должны поддерживать совместимость между версиями.

В этой главе важны два слова:

- encoding - превращение структуры данных в байты;
- decoding - восстановление структуры данных из байтов.

В памяти программа работает со структурами, объектами, map-ами, списками. Для
записи в файл, отправки по сети или сохранения в БД эти данные нужно
закодировать.

```text
in-memory object
  -> encode
       -> bytes on disk / network / queue
  -> decode
       -> in-memory object
```

## 1. Зачем вообще думать о форматах

Пока один процесс пишет и тут же читает свои данные, формат кажется технической
деталью. Но в реальных системах данные пересекают границы:

- процесс -> файл;
- сервис -> сервис;
- приложение -> database;
- producer -> message broker -> consumer;
- старая версия сервиса -> новая версия сервиса;
- backend -> mobile app, который пользователь обновит неизвестно когда.

Как только есть граница, появляется вопрос совместимости. Можно ли новая версия
прочитать старые данные? Можно ли старая версия проигнорировать новое поле?
Можно ли развернуть сервис постепенно, а не одновременно во всех местах?

## 2. Backward и forward compatibility

Backward compatibility:

```text
new code can read data written by old code
```

Новая версия умеет читать старый формат.

Forward compatibility:

```text
old code can read data written by new code
```

Старая версия не ломается, увидев данные от новой версии.

В rolling deploy нужны оба свойства. Пока часть instances уже обновилась, а
часть еще нет, они могут обмениваться данными в обе стороны.

Пример изменения:

```json
{
  "name": "Alice"
}
```

Новая версия добавляет поле:

```json
{
  "name": "Alice",
  "timezone": "Europe/Moscow"
}
```

Backward compatibility: новый код должен уметь прочитать старую запись без
`timezone`, например подставить default.

Forward compatibility: старый код должен не упасть из-за неизвестного поля
`timezone`.

## 3. Language-specific encoding

Многие языки имеют встроенную сериализацию объектов. Это удобно, но DDIA
показывает, почему для долгоживущих данных и межсервисного обмена это часто
плохая основа.

Проблемы:

- формат привязан к конкретному языку и runtime;
- сложно читать данные из другого языка;
- восстановление объектов может быть security risk;
- версионирование классов/типов становится хрупким;
- схема данных не всегда явно видна;
- производительность и размер могут быть хуже специализированных форматов.

Для хранения и обмена между системами лучше использовать языконезависимые
форматы с понятными правилами эволюции.

## 4. Text formats: JSON, XML, CSV

JSON, XML и CSV популярны, потому что их легко читать человеку и они широко
поддерживаются.

Сильные стороны:

- language-independent;
- удобны для HTTP APIs и конфигов;
- легко смотреть руками;
- много инструментов.

Но есть ограничения.

JSON и XML не всегда однозначно различают типы чисел. Большие integer values
могут потерять точность в окружениях, где числа представлены как floating point.

JSON не имеет встроенного типа для binary data, поэтому байты часто кодируют
через base64, увеличивая размер.

CSV еще беднее: нет схемы, типы полей не описаны, escaping и разделители часто
становятся источником несовместимости.

Вывод главы не в том, что text formats плохие. Они удобны, но совместимость и
типы приходится контролировать отдельно.

## 5. Binary encoding

Binary formats обычно компактнее и быстрее text formats, потому что не тратят
место на имена полей, кавычки, разделители и десятичную запись чисел.

Но просто "сжать JSON в байты" недостаточно. Важный вопрос - есть ли схема и
как она эволюционирует.

Семейство schema-based binary formats:

- Thrift;
- Protocol Buffers;
- Avro.

Общая идея: структура данных описывается схемой, а encoded data хранит значения
в компактном виде. Схема помогает понять, какие поля есть, какие у них типы и
как старые/новые версии должны взаимодействовать.

## 6. Thrift и Protocol Buffers

Thrift и Protocol Buffers используют schema definition. Полям назначаются
числовые tags. В binary data обычно записываются tag и value, а не полное имя
поля.

Упрощенно:

```text
schema:
  1: name string
  2: age int32

encoded:
  field 1 -> "Alice"
  field 2 -> 30
```

Почему tag важен: имя поля в коде можно изменить, но номер поля становится
частью wire format. Старый и новый код должны договориться, что field `1` все
еще означает тот же смысл.

Правила эволюции обычно такие:

- можно добавлять новое поле, если старый код умеет его пропустить;
- нельзя переиспользовать старый tag для нового смысла;
- удаленное поле лучше зарезервировать, чтобы случайно не вернуть tag;
- required fields опасны для эволюции, потому что старые данные их не содержат;
- optional/default values помогают читать старые записи.

Главная идея: совместимость достигается дисциплиной схемы, а не магией формата.

## 7. Avro

Avro тоже использует схему, но делает другой акцент. Encoded data может не
содержать field tags в том же смысле, поэтому чтение зависит от знания схемы.

В Avro важна пара:

- writer schema - схема, с которой данные были записаны;
- reader schema - схема, с которой данные читает текущий код.

При чтении Avro сопоставляет writer schema и reader schema. Это позволяет
обрабатывать изменения: новые поля с default values, удаленные поля,
переименования через aliases.

Avro особенно естественен там, где схема хранится рядом с большим набором
записей или в schema registry: например в файлах данных, event streams и
аналитических pipelines.

## 8. Schema evolution

Schema evolution - это правила изменения схемы без поломки старого и нового
кода.

Типовые безопасные изменения:

- добавить optional field;
- добавить поле с default value;
- удалить поле, если reader умеет жить без него;
- расширить enum только если старый код корректно обрабатывает неизвестные
  значения;
- не менять смысл существующего поля;
- не переиспользовать идентификатор поля.

Опасные изменения:

- сделать optional field required;
- изменить тип поля несовместимо;
- переиспользовать tag/field id;
- начать писать значение, которое старый reader не ожидает;
- удалить поле, которое старый код считает обязательным.

Совместимость - это свойство пары версий, а не просто красивый формат.

## 9. Databases и эволюция схемы

База данных хранит данные дольше, чем живет один deploy приложения. Поэтому
старые и новые версии схемы могут сосуществовать.

Реляционная схема меняется через migrations:

```text
old app + old schema
new schema supports old app
new app writes new fields
old fields removed later
```

Безопасный rollout часто делают в несколько шагов:

1. Добавить новое nullable/default поле.
2. Развернуть код, который умеет читать старую и новую форму.
3. Backfill старые данные.
4. Переключить writes на новую форму.
5. Удалить старое поле только после того, как старый код точно исчез.

Document databases дают больше гибкости: разные документы могут иметь разные
поля. Но ответственность переезжает в приложение: код должен уметь читать
старые варианты документов и мигрировать их явно или лениво при чтении.

## 10. Dataflow через databases

Один процесс пишет в БД, другой читает из БД. Это тоже форма dataflow.

Совместимость нужна, потому что writer и reader могут быть разными версиями:

```text
service A v2 writes new field
service B v1 reads same table later
```

Если `B v1` ломается от нового поля или нового значения enum, rolling deploy
становится опасным. Если `A v2` перезаписывает запись и теряет неизвестные ему
поля, можно случайно стереть данные новой версии.

Практический вывод: read-modify-write должен быть аккуратным. Старый код,
который не знает новое поле, не должен уничтожать его при сохранении объекта.

## 11. Dataflow через services

В service-oriented architecture или microservices один сервис вызывает другой
через API. Формат может быть JSON over HTTP, RPC, gRPC, Thrift, Protocol
Buffers или что-то свое.

Проблема та же: клиенты и серверы обновляются независимо.

Нужно понимать:

- кто контролирует клиентов;
- можно ли заставить всех обновиться быстро;
- есть ли mobile/desktop clients с долгим жизненным циклом;
- поддерживаются ли несколько версий API;
- что происходит с неизвестными полями;
- как добавляются новые обязательные параметры.

Backward compatibility для API часто важнее, чем для внутренних storage
formats, потому что внешних клиентов нельзя обновить синхронно с сервером.

## 12. REST и RPC

DDIA сравнивает разные стили service communication.

REST строится вокруг ресурсов, HTTP-методов и стандартной инфраструктуры Web.
Он хорошо работает там, где важны compatibility, кэширование, proxies,
debuggability и независимые клиенты.

RPC старается сделать удаленный вызов похожим на локальный вызов функции. Это
удобно, но может скрывать важную разницу: сеть ненадежна, вызов может
таймаутиться, выполниться частично, выполниться дважды или вернуть ответ после
повтора.

Главный урок: remote call не равен local function call. Протокол должен
учитывать latency, retries, idempotency, partial failures и evolution.

## 13. Asynchronous message passing

Еще один dataflow - через message broker:

```text
producer -> queue/topic -> consumer
```

Сообщение может жить дольше процесса producer. Consumer может быть старой или
новой версией. Очередь может накопить backlog, и новая версия consumer будет
читать сообщения, записанные старой версией producer.

Отсюда требования:

- сообщение должно быть self-contained или иметь доступную schema;
- consumer должен уметь игнорировать неизвестные поля;
- producer не должен резко менять смысл полей;
- schema registry может помогать проверять совместимость;
- retry и dead-letter queue должны учитывать старые форматы.

Асинхронность снижает связность по времени, но усиливает требования к
совместимости формата.

## 14. Rolling upgrades

Глава постоянно возвращает к одной практической картине: система обновляется
постепенно.

```text
time T0: all services v1
time T1: some services v2, some services v1
time T2: all services v2
time T3: old data from v1 still exists
```

В момент `T1` нужна совместимость между версиями кода. В момент `T3` новая
версия все еще должна читать старые данные.

Поэтому безопасное изменение формата почти всегда staged:

1. Read old and new.
2. Write old-compatible format.
3. Roll out all readers.
4. Start writing new format.
5. Later remove old compatibility.

Это применимо к database schema, API payloads и messages.

## 15. Практическая модель главы

Можно свести главу к цепочке вопросов:

1. Где данные кодируются в байты?
2. Кто потом будет их читать?
3. Могут ли writer и reader быть разными версиями?
4. Нужно ли читать старые данные новым кодом?
5. Нужно ли старому коду пережить новые данные?
6. Есть ли schema и где она хранится?
7. Какие изменения схемы безопасны?
8. Как пройдет rolling deploy?
9. Что будет с queued messages и archived data?
10. Не потеряет ли старый код неизвестные поля при перезаписи?

Если на эти вопросы нет ответа, изменение схемы может выглядеть маленьким в
коде, но стать большим production-risk.

## Ответ на собеседовании

Encoding and Evolution - это глава про совместимость данных во времени. Данные
живут на диске, в БД, в очередях и между сервисами дольше, чем один deploy.
Поэтому формат должен позволять старому и новому коду сосуществовать. Backward
compatibility означает, что новый код читает старые данные; forward
compatibility - что старый код не ломается от новых данных. JSON/XML/CSV удобны
и читаемы, но слабее по типам и схеме. Thrift, Protocol Buffers и Avro дают
schema-based binary encoding, но требуют дисциплины evolution: не
переиспользовать field ids, добавлять optional/default поля, осторожно удалять
и менять типы. Главный практический вывод - любые изменения формата нужно
проектировать под rolling deploy, старые данные, очереди и независимое обновление
клиентов.
