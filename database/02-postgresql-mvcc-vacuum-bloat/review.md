> Что физически происходит с данными, когда мы пишем, обновляем и удаляем строки, почему место не освобождается сразу, почему таблицы и индексы распухают, зачем нужен VACUUM, когда нужен REINDEX и как это все превращается в бизнесовые проблемы.

## Главная мысль

PostgreSQL не обновляет строку "на месте" в привычном смысле.

Когда ты делаешь:

```sql
UPDATE users
SET name = 'Vlad'
WHERE id = 10;
```

PostgreSQL обычно:

1. Оставляет старую версию строки в таблице.
2. Помечает старую версию как больше не актуальную для новых транзакций.
3. Создает новую версию строки.
4. При необходимости обновляет индексы.
5. Позже, когда старую версию уже точно никто не может видеть, `VACUUM` может убрать ее физически или пометить место как пригодное для повторного использования.

Именно из-за этого:

- чтение не блокирует запись так грубо, как в некоторых lock-based моделях;
- старые транзакции могут видеть старую версию данных;
- появляются dead tuples;
- нужен VACUUM;
- таблицы и индексы могут раздуваться;
- долгие транзакции становятся опасными;
- частые `UPDATE` могут быть дороже, чем кажется;
- индекс на часто изменяемой колонке может стать источником write amplification.

Короткая формулировка для интервью:

> PostgreSQL использует MVCC: каждая транзакция видит снимок данных, а изменения создают новые версии строк. Старые версии остаются в heap до тех пор, пока могут быть видны старым транзакциям. Потом VACUUM очищает dead tuples, обновляет карты видимости и свободного места, а при необходимости нужно бороться с bloat через настройки, изменение паттерна записи, VACUUM FULL, REINDEX или partition maintenance.

## Простое введение на бытовом примере

Представь таблицу:

```sql
CREATE TABLE accounts (
	id bigint PRIMARY KEY,
	balance numeric NOT NULL,
	status text NOT NULL,
	updated_at timestamptz NOT NULL
);
```

Есть строка:

```text
id = 1, balance = 100, status = active, updated_at = 10:00
```

Транзакция A начинает читать:

```sql
BEGIN;

SELECT balance
FROM accounts
WHERE id = 1;
```

Она увидела `balance = 100`.

Параллельно транзакция B делает:

```sql
UPDATE accounts
SET balance = 80,
    updated_at = now()
WHERE id = 1;

COMMIT;
```

Что должна видеть транзакция A?

Ответ зависит от уровня изоляции:

- в `READ COMMITTED` следующий statement транзакции A может увидеть уже `80`;
- в `REPEATABLE READ` транзакция A продолжит видеть старый снимок, то есть `100`;
- в `SERIALIZABLE` PostgreSQL еще дополнительно следит за конфликтами сериализации.

Чтобы это работало, PostgreSQL не может просто стереть `100` и заменить на `80`. Кому-то еще может быть нужна версия `100`.

Поэтому появляются две физические версии:

```text
old version: balance = 100
new version: balance = 80
```

Для новых транзакций видима новая. Для старых снимков может быть видима старая.

Позже, когда не останется транзакций, которым нужна старая версия, `VACUUM` сможет ее убрать.

## Бизнесовый смысл MVCC

MVCC нужен не ради внутренней красоты.

Он позволяет бизнес-системе:

- читать данные во время записи;
- выполнять много пользовательских запросов параллельно;
- не блокировать весь сервис при каждом `UPDATE`;
- давать каждой транзакции консистентную картину данных;
- откатывать транзакции;
- поддерживать уровни изоляции;
- не показывать пользователю частично записанные изменения.

Но цена:

- больше физических записей;
- больше WAL;
- больше работы для индексов;
- dead tuples;
- autovacuum;
- bloat;
- риск деградации из-за long-running transactions;
- необходимость мониторить не только SQL, но и "гигиену" базы.

На интервью это можно выразить так:

> MVCC снижает блокировки между читателями и писателями, но переносит часть сложности в хранение версий и фоновую очистку. Поэтому в PostgreSQL производительность зависит не только от индексов, но и от того, насколько здорово работает VACUUM и нет ли паттернов, которые постоянно создают мусорные версии строк.

## Физическая база: relation, page, tuple

В PostgreSQL таблица и индекс - это relation.

Relation физически хранится в файлах. Файл разбит на страницы, обычно по 8 KB.

Табличные данные лежат в heap. Heap - это набор страниц, внутри которых лежат tuple, то есть версии строк.

Упрощенно:

```text
table accounts
  heap file
    page 0
      tuple version
      tuple version
    page 1
      tuple version
      tuple version
```

Индекс хранится отдельно:

```text
index accounts_pkey
  btree pages
    key id=1 -> TID(page=123, offset=5)
```

TID - физическая ссылка на tuple в heap.

Очень важно:

> Индекс обычно указывает не на "логическую строку id=1", а на конкретную физическую версию строки в heap.

Поэтому при обычном `UPDATE`, если создается новая версия строки, индексам может понадобиться новая запись, которая указывает на новый TID.

## Tuple header: xmin, xmax, ctid

Каждая версия строки в PostgreSQL имеет служебные поля. На интервью не нужно знать все флаги tuple header, но базовые поля важно понимать.

Главные:

- `xmin` - ID транзакции, которая создала эту версию;
- `xmax` - ID транзакции, которая удалила или заменила эту версию, если такое произошло;
- `ctid` - физическое расположение tuple; при update может указывать на следующую версию в цепочке;
- служебные флаги видимости и статуса.

Упрощенный пример:

```text
version 1:
  id = 1
  balance = 100
  xmin = 1000
  xmax = 1010
  ctid = (5, 1)

version 2:
  id = 1
  balance = 80
  xmin = 1010
  xmax = 0
  ctid = (8, 4)
```

Смысл:

- версия 1 создана транзакцией `1000`;
- версия 1 заменена транзакцией `1010`;
- версия 2 создана транзакцией `1010`;
- версия 2 пока актуальная, потому что `xmax = 0`.

На самом деле видимость определяется не только числами `xmin/xmax`, а еще статусом транзакций: committed, aborted, in progress. Эти статусы PostgreSQL хранит отдельно.

Но для интервью достаточно модели:

> `xmin` говорит, кто создал версию, `xmax` говорит, кто ее удалил или заменил, а snapshot транзакции решает, какие версии строки видимы.

## Snapshot: что видит транзакция

Snapshot - это снимок видимости транзакций на определенный момент.

Он отвечает на вопрос:

> Какие transaction IDs уже зафиксированы, какие еще активны, какие начались позже моего снимка?

Когда PostgreSQL читает tuple, он проверяет:

- транзакция, создавшая tuple, уже закоммичена?
- была ли она закоммичена до моего snapshot?
- не была ли tuple удалена или заменена транзакцией, которая для меня уже видима?
- не была ли удаляющая транзакция откатана?

Упрощенная логика:

```text
tuple visible if:
  creating transaction committed and visible to snapshot
  and deleting/updating transaction either absent, aborted, or not visible to snapshot
```

Именно snapshot позволяет старой транзакции видеть старую версию строки, хотя новая версия уже закоммичена.

## Уровни изоляции и MVCC

В PostgreSQL важно понимать связь MVCC с уровнями изоляции.

### READ COMMITTED

Это дефолтный уровень.

Каждый statement видит новый snapshot.

Пример:

```sql
BEGIN;

SELECT balance FROM accounts WHERE id = 1; -- увидел 100

-- другая транзакция обновила balance до 80 и закоммитила

SELECT balance FROM accounts WHERE id = 1; -- может увидеть 80

COMMIT;
```

Для большинства OLTP-приложений это нормальный уровень, но нужно помнить: внутри одной транзакции два `SELECT` могут увидеть разные committed-данные.

### REPEATABLE READ

Транзакция видит один snapshot на протяжении всей транзакции.

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;

SELECT balance FROM accounts WHERE id = 1; -- 100

-- другая транзакция обновила до 80 и закоммитила

SELECT balance FROM accounts WHERE id = 1; -- все еще 100

COMMIT;
```

Это удобно для консистентных отчетов, но такие транзакции опасны, если живут долго: они могут удерживать старый horizon видимости и мешать VACUUM удалять старые версии.

### SERIALIZABLE

PostgreSQL пытается обеспечить результат, эквивалентный последовательному выполнению транзакций.

Это не "просто больше локов". PostgreSQL использует Serializable Snapshot Isolation и может завершить транзакцию ошибкой сериализации, если обнаружит опасный конфликт.

В приложении это значит:

> На SERIALIZABLE нужно быть готовым ретраить транзакцию.

## Что происходит при INSERT

Команда:

```sql
INSERT INTO accounts(id, balance, status, updated_at)
VALUES (1, 100, 'active', now());
```

Упрощенная физическая цепочка:

1. PostgreSQL выбирает страницу heap, где есть место.
2. Добавляет новую tuple version.
3. Ставит `xmin` равным текущей транзакции.
4. Пишет WAL-запись, чтобы изменение можно было восстановить после crash.
5. Добавляет записи во все индексы таблицы.
6. Буфер страницы становится dirty.
7. При commit записывается commit record в WAL.
8. Позже dirty page будет сброшена на диск checkpointer/background writer.

Важный момент:

> Commit не обязательно означает, что heap-страница уже физически записана на диск. Надежность обеспечивается WAL: сначала журнал, потом страницы данных.

На запись влияют:

- количество индексов;
- тип индексов;
- размер строки;
- наличие TOAST;
- заполненность страниц;
- WAL bandwidth;
- fsync;
- checkpoints;
- триггеры и constraints.

## Что происходит при SELECT

Запрос:

```sql
SELECT *
FROM accounts
WHERE id = 1;
```

Через primary key:

1. PostgreSQL идет в B-tree индекс.
2. Находит TID подходящей tuple.
3. Идет в heap.
4. Проверяет видимость tuple для snapshot.
5. Если tuple видима, возвращает строку.
6. Если tuple не видима, может идти по HOT-chain или искать другую версию, в зависимости от плана и структуры.

Почему индекс не гарантирует "прочитал только индекс и все":

- нужно проверить MVCC-видимость;
- нужны колонки, которых нет в индексе;
- страница может быть не all-visible;
- index scan часто требует heap fetch.

Отсюда связь с VACUUM:

> Когда VACUUM помечает страницы как all-visible в visibility map, PostgreSQL чаще может делать Index Only Scan без похода в heap.

## Hint bits

Когда PostgreSQL проверяет видимость tuple, ему иногда нужно узнать статус транзакции из служебных структур: committed, aborted, in progress.

Чтобы не делать эту проверку снова и снова, PostgreSQL может записывать в tuple header специальные hint bits.

Упрощенно:

```text
эта tuple создана уже committed-транзакцией
эта tuple удалена aborted-транзакцией
```

Это ускоряет будущие проверки видимости.

Интересный нюанс:

> Даже SELECT иногда может немного менять страницу, выставляя hint bits.

Это не логическое изменение пользовательских данных, но физическая страница может стать dirty. Поэтому в PostgreSQL чтение не всегда абсолютно "read-only" на уровне файловых страниц.

Для интервью достаточно понимать:

- MVCC visibility check не бесплатный;
- PostgreSQL кеширует часть результатов через hint bits;
- это еще одна причина, почему heap access важен даже при index scan.

## Что происходит при DELETE

Команда:

```sql
DELETE FROM accounts
WHERE id = 1;
```

PostgreSQL обычно не стирает строку физически сразу.

Он:

1. Находит tuple.
2. Ставит в нее `xmax` текущей транзакции.
3. Пишет WAL.
4. После commit новые транзакции больше не видят эту tuple.
5. Старые транзакции могут продолжать ее видеть, если их snapshot старше удаления.
6. Позже VACUUM удалит dead tuple или пометит место как reusable.

Индексные записи при DELETE тоже не исчезают мгновенно. Они продолжают указывать на tuple, которая потом будет признана невидимой. Очисткой займется VACUUM.

Бизнесовый вывод:

> Массовый DELETE на большой таблице не обязательно сразу вернет место операционной системе. Он может создать огромный объем dead tuples, нагрузить WAL, индексы, autovacuum и ухудшить latency.

Часто вместо массового DELETE лучше:

- удалять батчами;
- использовать partitioning и `DROP PARTITION`;
- архивировать;
- заранее проектировать retention;
- следить за autovacuum;
- не делать один гигантский delete в рабочее время.

## Что происходит при UPDATE

Команда:

```sql
UPDATE accounts
SET balance = balance - 20,
    updated_at = now()
WHERE id = 1;
```

PostgreSQL обычно делает UPDATE как "delete old version + insert new version":

1. Находит старую tuple.
2. Ставит старой tuple `xmax`.
3. Создает новую tuple с новым `xmin`.
4. Связывает версии через `ctid`/heap chain.
5. Пишет WAL.
6. Обновляет индексы, если нужен обычный non-HOT update.
7. После commit новые транзакции видят новую версию.
8. VACUUM позже уберет старую.

Важно:

> UPDATE создает мусорные версии строк. Частые UPDATE на горячей таблице - один из главных источников bloat.

## Когда обновляются индексы

Это один из самых важных практических вопросов.

### INSERT

При `INSERT` PostgreSQL добавляет записи во все индексы таблицы.

Если таблица имеет 8 индексов, один insert потенциально должен затронуть 8 индексных структур.

### DELETE

При `DELETE` heap tuple помечается удаленной. Индексные записи обычно остаются до VACUUM. Индексные сканы будут находить кандидата, а потом проверять видимость в heap.

### UPDATE без HOT

При обычном update создается новая heap tuple. Так как индексные записи указывают на физический TID, для новой tuple нужны новые индексные записи.

Ключевой нюанс:

> Если HOT невозможен, PostgreSQL может добавлять новые записи во все индексы, даже если значения индексированных колонок логически не поменялись, потому что новая версия строки лежит по новому TID.

Это причина, почему "просто обновляем `updated_at`" может быть дорогим, если таблица имеет много индексов.

### UPDATE с изменением indexed column

Если меняется колонка, которая участвует в индексе, индекс точно должен отразить новое значение.

Пример:

```sql
CREATE INDEX idx_accounts_status ON accounts(status);

UPDATE accounts
SET status = 'blocked'
WHERE id = 1;
```

Индекс по `status` должен получить запись для нового статуса.

Старая запись физически будет очищена позже.

### UPDATE и partial index

Partial index:

```sql
CREATE INDEX idx_jobs_ready
ON jobs(created_at)
WHERE state = 'ready';
```

Если строка меняет `state`:

```sql
UPDATE jobs
SET state = 'done'
WHERE id = 10;
```

строка может выйти из partial index. Это тоже индексная работа.

Если строка меняет `state` с `done` на `ready`, она входит в partial index.

Partial index дешевле полного, но predicate тоже влияет на update.

### UPDATE и expression index

Expression index:

```sql
CREATE INDEX idx_users_lower_email
ON users (lower(email));
```

Если меняется `email`, PostgreSQL должен пересчитать выражение и обновить индексную структуру.

Если функция дорогая, update становится дороже.

### UPDATE и GIN

GIN-индексы по `jsonb`, массивам, full-text часто тяжелее на запись.

Пример:

```sql
CREATE INDEX idx_events_payload_gin
ON events USING gin(payload);
```

Если часто обновлять `payload`, можно получить:

- рост WAL;
- рост pending list;
- дорогие flush/cleanup;
- нестабильный latency commit;
- сильный write amplification.

GIN отлично помогает чтению по JSON/массивам, но на горячей таблице его цена должна быть осознанной.

## HOT updates

HOT - Heap-Only Tuple.

Это оптимизация PostgreSQL, которая позволяет при некоторых `UPDATE` не добавлять новые записи в индексы.

Упрощенная идея:

```text
index points to old heap tuple
old heap tuple points to newer heap tuple on same page
newer heap tuple points to newest heap tuple on same page
```

Индекс ведет к корню цепочки, а PostgreSQL внутри heap находит видимую версию.

HOT полезен, потому что:

- меньше индексных записей;
- меньше index bloat;
- дешевле update;
- проще cleanup внутри страницы;
- меньше WAL для индексов.

Основные условия HOT:

1. Изменяемые колонки не должны требовать обновления обычных индексов.
2. Новая версия строки должна помещаться на той же heap-странице.

Практическое правило:

> Если часто обновляемая колонка не участвует в индексах и на странице есть свободное место, PostgreSQL может сделать HOT update.

Пример хорошего кандидата:

```sql
CREATE TABLE sessions (
	id bigint PRIMARY KEY,
	user_id bigint NOT NULL,
	last_seen_at timestamptz NOT NULL
);

CREATE INDEX idx_sessions_user_id ON sessions(user_id);
```

Если часто обновлять только `last_seen_at`, и `last_seen_at` не индексирован:

```sql
UPDATE sessions
SET last_seen_at = now()
WHERE id = $1;
```

PostgreSQL может использовать HOT, если есть место на странице.

Если добавить индекс:

```sql
CREATE INDEX idx_sessions_last_seen_at
ON sessions(last_seen_at);
```

то каждое обновление `last_seen_at` будет требовать индексной работы. Это может сильно ухудшить запись.

На ревью:

> Если поле часто обновляется, надо очень осторожно индексировать его. Индекс может убить HOT updates и резко увеличить write amplification.

## Fillfactor

Fillfactor определяет, насколько плотно PostgreSQL заполняет страницы.

По умолчанию таблицы обычно заполняются довольно плотно. Если строка потом обновляется, новая версия может не поместиться на той же странице. Тогда HOT невозможен.

Можно оставить место на странице:

```sql
ALTER TABLE sessions SET (fillfactor = 80);
```

Это значит: при будущей упаковке/вставке PostgreSQL будет стремиться оставлять часть страницы свободной.

Плюсы:

- больше шансов на HOT;
- меньше index bloat на update-heavy таблицах;
- меньше page splits/перемещений.

Минусы:

- таблица занимает больше места;
- sequential scan читает больше страниц;
- не лечит уже заполненные страницы без rewrite.

Применять надо точечно:

- горячие update-heavy таблицы;
- часто меняющиеся статусы;
- сессии;
- очереди;
- таблицы с `last_seen`, `updated_at`, counters.

## FSM: Free Space Map

Free Space Map - карта свободного места.

Она помогает PostgreSQL понять, на каких страницах таблицы есть место для новых tuple.

Когда VACUUM освобождает место внутри страниц, он обновляет FSM. Потом INSERT/UPDATE может переиспользовать это место.

Важно:

> Обычный VACUUM чаще всего не уменьшает размер файла таблицы на диске. Он делает место внутри файла пригодным для повторного использования.

Пример:

1. Таблица была 100 GB.
2. Удалили 40 GB строк.
3. VACUUM прошел.
4. Файл таблицы все еще может быть около 100 GB.
5. Но внутри есть свободное место, куда будущие INSERT/UPDATE могут писать.

Если бизнес смотрит на диск и спрашивает "почему DELETE не освободил место", ответ:

> DELETE логически удалил строки, VACUUM сделал место reusable внутри таблицы, но операционной системе место обычно не возвращается без rewrite-операций вроде VACUUM FULL, CLUSTER, pg_repack или пересоздания/партиционирования.

## VM: Visibility Map

Visibility Map - карта видимости страниц.

Она хранит информацию:

- все tuple на странице видимы всем транзакциям;
- все tuple на странице frozen.

Зачем это нужно:

1. VACUUM может пропускать страницы, которые не требуют обработки.
2. Index Only Scan может не ходить в heap, если страница помечена all-visible.

Связь с производительностью чтения:

```sql
SELECT amount
FROM payments
WHERE user_id = $1
ORDER BY created_at DESC
LIMIT 20;
```

Даже если все нужные колонки есть в индексе, PostgreSQL может делать heap fetch, если visibility map не говорит, что heap-страница all-visible.

После нормального VACUUM больше страниц могут стать all-visible, и index-only scan может стать эффективнее.

## WAL: почему commit быстрый и надежный

WAL - Write-Ahead Log.

Идея:

> Сначала надежно записываем описание изменения в журнал, потом когда-нибудь записываем сами страницы данных.

Если сервер упадет:

- PostgreSQL поднимется;
- прочитает WAL;
- повторит нужные изменения;
- восстановит консистентное состояние.

При `UPDATE` пишутся WAL-записи о heap-изменениях, index-изменениях и других действиях.

Чем больше индексов и чем тяжелее обновления, тем больше WAL.

Бизнесовые эффекты:

- репликация отстает, потому что надо передать и применить больше WAL;
- backups/PITR занимают больше места;
- диск с WAL забивается;
- commit latency растет при насыщении IO;
- массовые миграции создают огромный WAL storm.

На ревью миграции:

> Большой UPDATE/DELETE/CREATE INDEX - это не только нагрузка на таблицу. Это еще WAL, репликация, checkpoints, autovacuum и диск.

## Shared buffers, OS cache и dirty pages

PostgreSQL читает и пишет страницы через shared buffers.

Упрощенно:

1. Запросу нужна страница.
2. PostgreSQL ищет ее в shared buffers.
3. Если нет, читает с диска или из OS page cache.
4. При изменении страница в shared buffers становится dirty.
5. WAL должен быть записан до сброса dirty page.
6. Background writer и checkpointer постепенно сбрасывают dirty pages.

Важно различать:

- shared buffers - кэш страниц внутри PostgreSQL;
- OS page cache - кэш операционной системы;
- work_mem - память на сортировки/hash operations одного узла плана;
- maintenance_work_mem - память на VACUUM, CREATE INDEX и обслуживание;
- autovacuum_work_mem - память для autovacuum workers.

Типичная ошибка:

> Думать, что если запрос второй раз быстрый, индекс идеален. Возможно, данные просто оказались в cache.

В диагностике поэтому смотрят:

```sql
EXPLAIN (ANALYZE, BUFFERS)
...
```

`BUFFERS` помогает понять, сколько было hit/read/dirtied/written.

## Checkpoint и background writer

Dirty pages не сбрасываются на диск строго при каждом commit. Это было бы слишком дорого.

PostgreSQL использует:

- WAL для надежности;
- background writer для постепенной записи dirty pages;
- checkpoint, чтобы ограничить объем WAL, который нужно проигрывать при recovery.

Checkpoint говорит:

> До этой точки все нужные dirty pages должны быть записаны так, чтобы recovery мог стартовать отсюда.

Если checkpoints слишком частые или слишком резкие:

- latency может прыгать;
- IO может уходить в полку;
- запись становится нестабильной.

Массовые UPDATE/DELETE/CREATE INDEX могут усиливать checkpoint pressure.

На интервью достаточно сказать:

> PostgreSQL пишет изменения через WAL, а страницы данных сбрасываются асинхронно. Поэтому производительность записи зависит не только от SQL, но и от WAL, dirty buffers, checkpoints и фоновой записи.

## Dead tuples

Dead tuple - версия строки, которая больше не видима ни одной активной транзакции и может быть удалена.

Но есть похожие состояния:

- live tuple - актуальная и видимая версия;
- recently dead - версия уже удалена/заменена, но может быть видима старому snapshot;
- dead - точно никому не нужна;
- frozen - старая tuple помечена так, чтобы не зависеть от старого transaction ID.

Почему dead tuples не удаляются сразу?

Потому что при DELETE/UPDATE PostgreSQL не знает, нет ли старой транзакции, которая еще должна видеть старую версию.

Пример:

```sql
-- transaction A
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM accounts WHERE id = 1; -- видит old version

-- transaction B
UPDATE accounts SET balance = 80 WHERE id = 1;
COMMIT;

-- transaction A все еще открыта
```

Пока A открыта, старая версия может быть нужна. VACUUM не может ее удалить.

Так появляются проблемы с long-running transactions.

## OldestXmin: горизонт очистки

VACUUM не решает самовольно, что старую версию можно удалить. Ему нужен горизонт видимости.

Упрощенно этот горизонт можно назвать `OldestXmin`:

> Самая старая транзакция/snapshot, из-за которой старые версии строк еще потенциально могут быть видимы.

Если tuple удалена транзакцией `2000`, но в системе есть транзакция со snapshot, который еще может видеть состояние до `2000`, tuple нельзя физически убрать.

Что может удерживать горизонт:

- обычная долгая транзакция;
- `idle in transaction`;
- долгий read-only отчет;
- prepared transaction;
- replication slot;
- logical decoding;
- некоторые служебные процессы.

Это объясняет важную мысль:

> VACUUM может запускаться регулярно, но не иметь права удалить мусор, пока кто-то держит старый snapshot.

## Page pruning

Не вся очистка ждет полноценного VACUUM.

PostgreSQL может выполнять page pruning при обычном доступе к heap-странице. Если backend читает или модифицирует страницу и видит, что внутри есть очевидно ненужные версии в HOT-chain, он может почистить часть цепочки.

Упрощенно:

```text
index -> tuple A -> tuple B -> tuple C

если A и B уже никому не нужны,
можно сократить/почистить chain,
оставив путь к актуальной версии
```

Page pruning помогает:

- уменьшить мусор внутри страницы;
- поддерживать HOT chains короче;
- освободить место на странице;
- снизить будущую работу VACUUM.

Но pruning не заменяет VACUUM:

- он локален для страницы;
- не обязан пройти всю таблицу;
- не выполняет всю работу по индексам;
- не занимается freeze в полном смысле;
- не обновляет все глобальные карты и статистику как regular vacuum.

На интервью:

> В PostgreSQL есть не только VACUUM. Часть локальной очистки heap-страниц может происходить при обычном доступе через page pruning, особенно для HOT chains. Но полноценная уборка dead tuples, индексов, visibility/free space maps и freeze - это задача VACUUM.

## VACUUM: что он делает

Обычный `VACUUM`:

```sql
VACUUM accounts;
```

Основные задачи:

1. Найти dead tuples.
2. Удалить или пометить их место как reusable.
3. Очистить индексные записи, которые ссылаются на dead tuples.
4. Обновить Free Space Map.
5. Обновить Visibility Map.
6. Заморозить старые tuple, если нужно.
7. Помочь index-only scans.
8. Обновить часть статистики обслуживания.

Важно:

> Обычный VACUUM не блокирует обычные SELECT/INSERT/UPDATE/DELETE так грубо, как VACUUM FULL, и обычно работает фоном через autovacuum.

Но VACUUM потребляет ресурсы:

- CPU;
- IO;
- память;
- может конкурировать с пользовательской нагрузкой;
- может создавать дополнительный WAL;
- на больших таблицах может идти долго.

## Фазы VACUUM упрощенно

Внутренние детали меняются между версиями PostgreSQL, но практическая модель такая.

### 1. Скан heap

VACUUM проходит по страницам таблицы и ищет tuple, которые можно удалить.

Он учитывает:

- видимость tuple;
- горизонт `OldestXmin`;
- visibility map;
- необходимость freeze;
- есть ли страницы, которые можно пропустить.

### 2. Сбор ссылок на dead tuples

VACUUM собирает TID мертвых tuple, чтобы потом почистить индексы.

Почему это нужно:

> Индексы указывают на физические версии строк. Если версия строки умерла, индексные записи, которые на нее ссылаются, тоже нужно убрать.

### 3. Index vacuum

VACUUM проходит по индексам и удаляет записи, которые указывают на dead heap tuples.

Чем больше индексов на таблице, тем дороже эта фаза.

Отсюда практический вывод:

> Лишние индексы вредят не только insert/update/delete, но и VACUUM.

### 4. Heap cleanup

После очистки индексов VACUUM может окончательно пометить dead tuple space в heap как reusable.

Внутри страницы место может быть освобождено и учтено в FSM.

### 5. Visibility map и freeze

Если на странице все tuple видимы всем, страница может получить all-visible bit.

Если tuple достаточно старые и заморожены, страница может получить all-frozen bit.

Это важно для:

- будущих VACUUM;
- index-only scans;
- защиты от transaction ID wraparound.

### 6. Truncate tail pages, если возможно

Иногда VACUUM может вернуть операционной системе страницы в конце relation, если они полностью свободны.

Но это не гарантируется и не является основным механизмом уменьшения размера таблицы.

Именно поэтому после большого DELETE обычный VACUUM часто не уменьшает файл.

## Почему обычный VACUUM не сжимает таблицу как архиватор

Наивное ожидание:

```text
было 100 GB
удалили 50 GB
VACUUM
стало 50 GB
```

В PostgreSQL обычно не так.

Причина в физической модели:

- живые tuple имеют физические адреса;
- индексы ссылаются на TID;
- другие tuple/HOT chains могут ссылаться внутри heap;
- свободное место может быть разбросано по страницам;
- перемещение живых tuple между страницами потребовало бы массового обновления индексов и сложной координации.

Обычный VACUUM поэтому делает более дешевую операцию:

```text
эта область внутри существующего файла свободна,
ее можно использовать для будущих insert/update
```

Чтобы реально компактно переписать таблицу, нужна rewrite-операция:

- `VACUUM FULL`;
- `CLUSTER`;
- `pg_repack`;
- создание новой таблицы и swap;
- dump/restore;
- drop/truncate partition.

## VACUUM ANALYZE

```sql
VACUUM (ANALYZE) accounts;
```

`VACUUM` очищает физический мусор.

`ANALYZE` собирает статистику для планировщика.

Это разные задачи, но их часто запускают вместе.

Если после массовой загрузки данных запросы строят плохие планы, может помочь:

```sql
ANALYZE accounts;
```

Если после массового удаления много dead tuples:

```sql
VACUUM accounts;
```

Если нужно и то и другое:

```sql
VACUUM (ANALYZE) accounts;
```

## Autovacuum

Autovacuum - фоновый механизм, который сам запускает VACUUM и ANALYZE.

Он нужен, потому что руками чистить все таблицы невозможно.

Упрощенно autovacuum запускается, когда число измененных/dead tuples превышает пороги.

Типовая идея порога:

```text
threshold + scale_factor * table_rows
```

Например, если таблица огромная, стандартный scale factor может означать, что autovacuum придет слишком поздно.

Для больших горячих таблиц часто настраивают параметры точечно:

```sql
ALTER TABLE jobs SET (
	autovacuum_vacuum_scale_factor = 0.01,
	autovacuum_vacuum_threshold = 10000,
	autovacuum_analyze_scale_factor = 0.005,
	autovacuum_analyze_threshold = 10000
);
```

Смысл:

- чаще vacuum;
- чаще analyze;
- меньше накопление мусора;
- стабильнее планы.

Но слишком агрессивный autovacuum тоже может вредить, если забирает IO у приложения.

Нужно искать баланс.

## Почему autovacuum может "не справляться"

Причины:

1. Слишком большие таблицы и дефолтные scale factors.

   Autovacuum приходит поздно.

2. Слишком много dead tuples создается быстрее, чем vacuum успевает чистить.

   Например, job queue с постоянными update статусов.

3. Долгие транзакции держат старый xmin.

   VACUUM проходит, но не может удалить recently dead tuples.

4. `idle in transaction`.

   Приложение открыло транзакцию и ничего не делает.

5. Replication slot удерживает старый xmin/catalog_xmin.

   Особенно опасно при отставшей логической репликации.

6. Autovacuum cost delay слишком ограничивает скорость.

   Он работает аккуратно, но слишком медленно для нагрузки.

7. Не хватает autovacuum workers.

   Много таблиц ждут обслуживания.

8. Долгие блокировки.

   Autovacuum может не получить нужные lock или быть отменен конфликтами.

9. Очень много индексов.

   Чистка индексов дороже.

## VACUUM FULL

```sql
VACUUM FULL accounts;
```

Это совсем другая операция.

Обычный VACUUM:

- чистит dead tuples;
- делает место reusable внутри файла;
- обычно не возвращает место OS;
- не требует полной эксклюзивной блокировки на все время как rewrite.

VACUUM FULL:

- переписывает таблицу в новый компактный файл;
- возвращает место операционной системе;
- требует тяжелую блокировку;
- может идти долго;
- создает много IO;
- требует дополнительное место на диске на время операции;
- может быть неприемлем в рабочее время.

На интервью:

> VACUUM FULL используют как тяжелую ремонтную операцию, когда нужно реально уменьшить физический размер таблицы. Для обычной регулярной очистки нужен autovacuum/regular vacuum, а не VACUUM FULL.

## ANALYZE

`ANALYZE` собирает статистику распределения данных.

```sql
ANALYZE orders;
```

Планировщик использует эту статистику для выбора:

- Seq Scan или Index Scan;
- Nested Loop или Hash Join;
- порядок join;
- использовать ли bitmap scan;
- сколько строк ожидать после filter.

После массовых изменений статистика может устареть.

Пример:

```sql
UPDATE orders
SET status = 'archived'
WHERE created_at < now() - interval '2 years';
```

Если после этого не обновить статистику, PostgreSQL может продолжить думать, что распределение `status` старое.

Autovacuum обычно запускает autoanalyze, но иногда нужно руками:

```sql
ANALYZE orders;
```

## Freeze и transaction ID wraparound

PostgreSQL transaction ID имеет ограниченный размер. Старые transaction IDs нельзя хранить бесконечно как обычные числа, потому что они могут "обернуться".

Чтобы этого не случилось, PostgreSQL делает freeze старых tuple.

Freeze означает:

> Эта tuple настолько старая и точно видимая всем, что ее можно считать созданной "до всех актуальных транзакций".

Если freeze не происходит, база приближается к transaction ID wraparound. Это критическая аварийная зона.

PostgreSQL запускает anti-wraparound autovacuum даже если обычный autovacuum отключен.

Бизнесовый смысл:

- нельзя бездумно отключать autovacuum;
- долгие транзакции могут мешать продвижению горизонта;
- старые базы требуют мониторинга `age(datfrozenxid)` и `relfrozenxid`;
- anti-wraparound vacuum может внезапно прийти с тяжелой нагрузкой, если долго игнорировать проблему.

Пример мониторинга:

```sql
SELECT
	datname,
	age(datfrozenxid) AS xid_age
FROM pg_database
ORDER BY xid_age DESC;
```

По таблицам:

```sql
SELECT
	n.nspname AS schema_name,
	c.relname AS table_name,
	age(c.relfrozenxid) AS xid_age
FROM pg_class c
JOIN pg_namespace n ON n.oid = c.relnamespace
WHERE c.relkind IN ('r', 'm')
ORDER BY xid_age DESC
LIMIT 20;
```

## Long-running transactions

Одна из самых частых реальных проблем.

Транзакция открылась и долго живет:

```sql
BEGIN;
SELECT ...
-- приложение пошло во внешний сервис
-- пользователь думает
-- соединение висит
```

Пока транзакция держит старый snapshot, PostgreSQL не может удалить версии строк, которые потенциально видимы этому snapshot.

Симптомы:

- растет `n_dead_tup`;
- autovacuum запускается, но не очищает достаточно;
- таблицы растут;
- индексы растут;
- запросы становятся медленнее;
- replication lag растет;
- возрастает disk usage;
- появляются idle in transaction sessions.

Диагностика:

```sql
SELECT
	pid,
	usename,
	state,
	xact_start,
	now() - xact_start AS xact_age,
	query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY xact_start;
```

Особенно опасно:

```text
state = idle in transaction
```

Это значит, что клиент открыл транзакцию и сейчас ничего не выполняет, но транзакция все еще держит ресурсы.

Практики:

- не держать транзакцию вокруг внешних HTTP/gRPC вызовов;
- не держать транзакцию вокруг пользовательского ввода;
- ставить `idle_in_transaction_session_timeout`;
- ставить `statement_timeout`;
- ставить `lock_timeout`;
- в Go всегда `defer tx.Rollback()` и явный `Commit`;
- закрывать rows;
- не забывать `cancel()` у context;
- не делать огромные отчеты в одной долгой transaction snapshot на primary.

## Replication slots и xmin

Replication slots помогают репликации не потерять WAL.

Но они могут удерживать:

- WAL;
- catalog_xmin;
- иногда мешать очистке старых версий, особенно для logical replication.

Если consumer логической репликации отстал или умер, slot может держать старый horizon.

Симптомы:

- растет диск с WAL;
- VACUUM не может чистить некоторые данные;
- `pg_replication_slots` показывает старые `restart_lsn`, `xmin`, `catalog_xmin`;
- база пухнет.

Диагностика:

```sql
SELECT
	slot_name,
	slot_type,
	active,
	restart_lsn,
	xmin,
	catalog_xmin
FROM pg_replication_slots;
```

На интервью можно сказать:

> Long-running transactions и replication slots могут удерживать старые версии строк. Поэтому VACUUM - это не только "запустить команду", а еще вопрос горизонта видимости.

## Bloat

Bloat - физическое раздувание таблиц или индексов.

Есть два близких, но разных эффекта:

1. Внутри таблицы есть свободное место после VACUUM.

   Это не всегда плохо. Если таблица продолжит получать новые строки, место будет переиспользовано.

2. Таблица или индекс стали намного больше, чем нужно для текущих live data.

   Это уже bloat как проблема.

Пример:

```text
live data: 20 GB
table file: 120 GB
indexes: 200 GB
```

Причины:

- много UPDATE/DELETE;
- long transactions мешали vacuum;
- autovacuum был слишком слабым;
- частые update indexed columns;
- плохой fillfactor для update-heavy таблицы;
- массовые delete без partitioning;
- GIN/JSONB на горячей таблице;
- много дублирующих индексов;
- редкий vacuum на огромной таблице;
- page splits в B-tree;
- устаревшие версии index tuples.

Бизнесовые последствия:

- растет счет за диск;
- backups дольше и больше;
- restore дольше;
- репликация тяжелее;
- cache hit ratio хуже, потому что cache забит мусорными страницами;
- index scan читает больше страниц;
- latency API растет;
- vacuum работает дольше;
- миграции идут дольше.

## Table bloat

Table bloat связан с heap.

Обычный VACUUM может убрать dead tuples и сделать место reusable, но файл может остаться большим.

Когда это нормально:

- таблица циклически растет и переиспользует место;
- нагрузка стабильна;
- disk usage не растет бесконечно;
- latency нормальная.

Когда плохо:

- таблица после массового удаления осталась огромной;
- новые данные не переиспользуют старые страницы;
- seq scan стал читать слишком много;
- backup/restore страдают;
- диск заканчивается.

Варианты лечения:

- подождать, если место будет переиспользовано;
- `VACUUM` для reusable space;
- `VACUUM FULL` для физического shrink, но с тяжелой блокировкой;
- `CLUSTER`;
- `pg_repack`;
- partitioning;
- пересоздание таблицы через create-copy-swap;
- изменить паттерн обновлений;
- настроить autovacuum/fillfactor.

## Index bloat

Index bloat особенно неприятен, потому что:

- индексы должны быть компактными для быстрых traversals;
- раздутый индекс хуже помещается в cache;
- B-tree может иметь лишние страницы;
- запросы по индексу начинают читать больше;
- запись в индекс дороже.

Обычный VACUUM чистит dead index entries, но не всегда возвращает индекс к идеальному компактному виду.

Для сильного index bloat часто используют:

```sql
REINDEX INDEX idx_orders_user_id;
```

или на продакшене:

```sql
REINDEX INDEX CONCURRENTLY idx_orders_user_id;
```

Можно перестроить все индексы таблицы:

```sql
REINDEX TABLE orders;
```

или безопаснее для доступности, если поддерживается:

```sql
REINDEX TABLE CONCURRENTLY orders;
```

Практический нюанс:

- обычный REINDEX быстрее, но берет более жесткие блокировки;
- REINDEX CONCURRENTLY дольше и сложнее, но позволяет продолжать работу;
- REINDEX требует дополнительное место на диске;
- после сбоя concurrent reindex могут остаться служебные/invalid индексы, их надо проверять.

## REINDEX

REINDEX пересоздает индекс.

Когда нужен:

- индекс сильно раздут;
- индекс поврежден;
- сменились настройки collation;
- после большого churn индекс стал слишком большим;
- нужно восстановить компактность;
- btree dedup/новая версия PostgreSQL может дать выигрыш после rebuild.

Команды:

```sql
REINDEX INDEX index_name;
REINDEX TABLE table_name;
REINDEX SCHEMA schema_name;
REINDEX DATABASE database_name;
```

Concurrent-варианты:

```sql
REINDEX INDEX CONCURRENTLY index_name;
REINDEX TABLE CONCURRENTLY table_name;
```

В продакшене обычно предпочитают concurrent, если downtime нельзя.

Но:

- concurrent дольше;
- больше IO;
- требует больше места;
- не запускается внутри обычного transaction block;
- нужно мониторить invalid indexes.

## CLUSTER

```sql
CLUSTER orders USING idx_orders_created_at;
```

CLUSTER физически переписывает таблицу в порядке индекса.

Плюсы:

- улучшает locality для range scans;
- может убрать table bloat;
- связанные данные лежат ближе физически.

Минусы:

- тяжелая операция;
- требует блокировки;
- порядок со временем снова деградирует при новых insert/update;
- нужно место и время.

Для OLTP используется осторожно.

## pg_repack

`pg_repack` - расширение, которое помогает перестроить таблицу/индексы с меньшими блокировками, чем `VACUUM FULL`.

Это не встроенная команда SQL PostgreSQL core, а extension/tool.

Плюсы:

- может вернуть место OS;
- меньше downtime;
- полезен для борьбы с bloat.

Минусы:

- нужно установить и поддерживать extension;
- нужны права;
- создает дополнительную нагрузку;
- требует место на диске;
- надо внимательно тестировать.

На интервью можно упомянуть:

> Если обычный VACUUM не возвращает место, а VACUUM FULL нельзя из-за блокировки, часто смотрят в сторону pg_repack или partition maintenance.

## Partitioning как способ избежать тяжелой очистки

Для больших таблиц с retention часто лучше не удалять старые строки миллионами, а партиционировать.

Пример:

```sql
CREATE TABLE events (
	id bigint,
	created_at timestamptz NOT NULL,
	payload jsonb NOT NULL
) PARTITION BY RANGE (created_at);
```

Тогда старые данные можно удалять так:

```sql
DROP TABLE events_2025_01;
```

или detach:

```sql
ALTER TABLE events DETACH PARTITION events_2025_01;
```

Это намного дешевле, чем:

```sql
DELETE FROM events
WHERE created_at < now() - interval '1 year';
```

Потому что массовый DELETE создает dead tuples и WAL, а drop partition убирает целый physical relation.

Бизнесовый вывод:

> Если retention известен заранее, partitioning часто является не оптимизацией, а способом избежать будущих аварий с VACUUM и bloat.

## Массовые UPDATE и DELETE

Большой антипаттерн:

```sql
UPDATE orders
SET status = 'archived'
WHERE created_at < now() - interval '2 years';
```

Если это миллионы строк:

- создаст миллионы новых версий;
- старые версии станут dead;
- обновит индексы;
- создаст много WAL;
- может вызвать replication lag;
- запустит autovacuum;
- может раздуть таблицу и индексы;
- может держать locks долго.

Лучше:

- батчить;
- ограничивать по primary key;
- делать паузы;
- мониторить replication lag;
- обновлять в непиковое время;
- заранее добавить нужные индексы;
- после операции `ANALYZE`;
- подумать, нужен ли вообще update старых строк;
- использовать partitioning, если это retention.

Пример батча:

```sql
WITH batch AS (
	SELECT id
	FROM orders
	WHERE created_at < now() - interval '2 years'
	  AND status <> 'archived'
	ORDER BY id
	LIMIT 5000
)
UPDATE orders o
SET status = 'archived'
FROM batch
WHERE o.id = batch.id;
```

Такую операцию запускают циклом из приложения/job, с паузами и мониторингом.

## Soft delete

Soft delete:

```sql
UPDATE documents
SET deleted_at = now()
WHERE id = $1;
```

Плюсы:

- можно восстановить;
- аудит;
- не ломаются ссылки;
- проще бизнес-логика удаления.

Минусы:

- это UPDATE, а не бесплатная пометка;
- старая версия становится dead;
- если `deleted_at` участвует в индексах, будет index churn;
- запросы должны всегда фильтровать `deleted_at IS NULL`;
- таблица физически растет;
- старые deleted rows мешают статистике и размеру;
- partial indexes становятся почти обязательными.

Хороший индекс:

```sql
CREATE INDEX idx_documents_alive_account_created
ON documents(account_id, created_at DESC)
WHERE deleted_at IS NULL;
```

Если soft-deleted данные нужно хранить долго, часто стоит:

- архивировать в отдельную таблицу;
- партиционировать по `deleted_at`/`created_at`;
- настроить retention;
- не держать все в одной горячей таблице.

## Job queues и state machines

Таблицы очередей часто выглядят так:

```sql
CREATE TABLE jobs (
	id bigserial PRIMARY KEY,
	state text NOT NULL,
	attempts int NOT NULL,
	available_at timestamptz NOT NULL,
	locked_at timestamptz,
	updated_at timestamptz NOT NULL
);
```

Типичные операции:

```sql
UPDATE jobs
SET state = 'processing',
    locked_at = now(),
    updated_at = now()
WHERE id = $1;
```

и потом:

```sql
UPDATE jobs
SET state = 'done',
    updated_at = now()
WHERE id = $1;
```

Это update-heavy workload.

Риски:

- много версий строк;
- state часто индексируют, и HOT невозможен;
- partial indexes входят/выходят;
- autovacuum должен работать активно;
- таблица быстро пухнет;
- индексы по state/available_at пухнут;
- долгие транзакции воркеров мешают cleanup.

Практики:

- partial index только по активным состояниям;
- батчи небольшие;
- `FOR UPDATE SKIP LOCKED`;
- не держать транзакцию во время выполнения внешней работы;
- после взятия job быстро commit;
- результат записывать отдельной короткой транзакцией;
- autovacuum per-table tuning;
- partition/archive done jobs.

## updated_at и last_seen_at

Очень частая бизнесовая мелочь, которая становится проблемой.

```sql
UPDATE users
SET last_seen_at = now()
WHERE id = $1;
```

Если это происходит на каждый запрос пользователя, таблица `users` становится write-heavy.

Если `last_seen_at` индексирован:

- каждый запрос пользователя обновляет индекс;
- HOT невозможен для этой колонки;
- растет WAL;
- растет bloat;
- страдает primary table.

Варианты:

- обновлять не чаще раза в N минут;
- хранить presence в Redis;
- писать события в append-only таблицу;
- агрегировать асинхронно;
- убрать индекс, если он не нужен;
- вынести часто меняемое поле в отдельную таблицу;
- fillfactor для таблицы.

На ревью:

> Часто обновляемые технические поля нельзя бездумно индексировать и обновлять на каждый request. Это может превратить read-heavy API в write-heavy нагрузку.

## Счетчики

Пример:

```sql
UPDATE posts
SET views_count = views_count + 1
WHERE id = $1;
```

Если делать на каждый просмотр:

- row-level lock на популярной строке;
- постоянные new tuple versions;
- bloat;
- WAL;
- contention;
- плохая масштабируемость.

Варианты:

- буферизовать счетчики;
- Redis/in-memory aggregation;
- append-only events + periodic aggregation;
- approximate counters;
- sharded counters;
- batch update.

## TOAST

TOAST - механизм PostgreSQL для хранения больших значений вне основной heap-страницы.

Например:

- большие `text`;
- большие `jsonb`;
- bytea;
- большие массивы.

Если строка содержит большой `jsonb`, он может лежать в TOAST-таблице.

Риски:

- update большого JSON может переписать TOAST-данные;
- GIN index по jsonb добавляет write cost;
- частые изменения одного поля внутри JSON не являются дешевыми patch-in-place;
- большие строки хуже помещаются на page;
- HOT может быть сложнее из-за размера новой версии.

На ревью:

> Если в OLTP-таблице часто обновляют большой jsonb, это может быть намного дороже, чем выглядит в коде. Нужно подумать о нормализации, отдельной таблице, event model или ограничении update-паттерна.

## Locks и MVCC

MVCC не означает, что locks не нужны.

PostgreSQL все равно использует locks:

- row locks при update/delete/select for update;
- table locks для DDL;
- predicate/SIREAD locks в Serializable;
- lightweight locks внутри движка;
- locks для индексов и буферов.

Пример:

```sql
UPDATE accounts
SET balance = balance - 100
WHERE id = 1;
```

Две транзакции, обновляющие одну и ту же строку, будут конфликтовать. Одна будет ждать другую.

MVCC помогает читателям не блокировать писателей в простых случаях, но writers одной строки все равно конфликтуют.

На интервью:

> MVCC не отменяет блокировки. Он уменьшает конфликты read/write, но write/write конфликты, DDL locks и explicit row locks остаются.

## Индексы и видимость

Индекс в PostgreSQL не хранит всю MVCC-истину.

Он может найти candidate tuples, но видимость проверяется через heap, кроме случаев index-only scan с visibility map.

Следствие:

1. Index Scan может делать много heap fetch.
2. Мертвые index entries могут ухудшать работу до VACUUM.
3. Index Only Scan зависит от visibility map.
4. Dead tuples в heap и index bloat связаны с качеством VACUUM.

Это объясняет, почему после VACUUM запросы иногда становятся быстрее даже без нового индекса.

## Когда обычный VACUUM не помогает вернуть место

Сценарий:

```sql
DELETE FROM events
WHERE created_at < now() - interval '1 year';

VACUUM events;
```

Почему файл не уменьшился?

Потому что обычный VACUUM:

- очищает dead tuples;
- обновляет FSM;
- позволяет переиспользовать место;
- но обычно не переписывает relation в компактный новый файл.

Он может вернуть OS место только в ограниченных случаях, например если свободные страницы оказались в конце relation и их можно обрезать. Но рассчитывать на это как на основной shrink-механизм нельзя.

Для реального уменьшения:

- `VACUUM FULL`;
- `CLUSTER`;
- `pg_repack`;
- dump/restore;
- create new table as + swap;
- drop/truncate partition.

## TRUNCATE

```sql
TRUNCATE TABLE events;
```

TRUNCATE быстро удаляет все строки и освобождает место.

Отличие от DELETE:

- не создает dead tuple для каждой строки;
- обычно намного быстрее;
- берет более сильную блокировку;
- удаляет все;
- может сбрасывать identity при `RESTART IDENTITY`;
- имеет особенности с foreign keys;
- транзакционный в PostgreSQL, то есть может быть rollback внутри транзакции.

Для очистки временных/буферных таблиц TRUNCATE часто лучше DELETE.

Но для частичного удаления не подходит.

## DROP PARTITION против DELETE

Если нужно удалить данные за месяц:

Плохо:

```sql
DELETE FROM events
WHERE created_at >= '2025-01-01'
  AND created_at < '2025-02-01';
```

Если это миллионы строк, будет много dead tuples и WAL.

Лучше при partitioning:

```sql
DROP TABLE events_2025_01;
```

или:

```sql
ALTER TABLE events DETACH PARTITION events_2025_01;
DROP TABLE events_2025_01;
```

Это удаляет целый physical relation.

## Наблюдаемость: что смотреть

### pg_stat_user_tables

```sql
SELECT
	schemaname,
	relname,
	n_live_tup,
	n_dead_tup,
	last_vacuum,
	last_autovacuum,
	vacuum_count,
	autovacuum_count,
	last_analyze,
	last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

Смотреть:

- много ли dead tuples;
- когда был autovacuum;
- происходит ли autoanalyze;
- есть ли таблицы, где autovacuum давно не был.

### pg_stat_all_tables HOT ratio

```sql
SELECT
	schemaname,
	relname,
	n_tup_upd,
	n_tup_hot_upd,
	CASE
		WHEN n_tup_upd = 0 THEN 0
		ELSE round(100.0 * n_tup_hot_upd / n_tup_upd, 2)
	END AS hot_update_percent
FROM pg_stat_user_tables
ORDER BY n_tup_upd DESC
LIMIT 20;
```

Если таблица много обновляется, но `n_tup_hot_upd` низкий, стоит проверить:

- не индексируем ли часто меняемые поля;
- хватает ли fillfactor;
- не слишком ли большие строки;
- есть ли место на страницах.

### pg_stat_user_indexes

```sql
SELECT
	schemaname,
	relname,
	indexrelname,
	idx_scan,
	idx_tup_read,
	idx_tup_fetch
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC;
```

Помогает искать:

- неиспользуемые индексы;
- индексы с большим read/fetch;
- кандидаты на удаление или reindex.

### Размеры таблиц и индексов

```sql
SELECT
	relname,
	pg_size_pretty(pg_relation_size(relid)) AS table_size,
	pg_size_pretty(pg_indexes_size(relid)) AS indexes_size,
	pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 20;
```

Если indexes_size сильно больше table_size, это повод смотреть:

- нет ли лишних индексов;
- нет ли index bloat;
- нет ли слишком широких INCLUDE;
- нет ли GIN на больших jsonb;
- насколько write-heavy таблица.

### pg_stat_progress_vacuum

```sql
SELECT *
FROM pg_stat_progress_vacuum;
```

Показывает текущий прогресс VACUUM.

Полезно, если autovacuum "висит" на большой таблице.

### pg_stat_activity

```sql
SELECT
	pid,
	state,
	wait_event_type,
	wait_event,
	xact_start,
	now() - xact_start AS xact_age,
	query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY xact_start;
```

Ищем:

- long-running transactions;
- idle in transaction;
- старые snapshots;
- блокировки.

### pg_locks

```sql
SELECT *
FROM pg_locks
WHERE NOT granted;
```

Для диагностики блокировок.

### Репликационные слоты

```sql
SELECT
	slot_name,
	slot_type,
	active,
	restart_lsn,
	xmin,
	catalog_xmin
FROM pg_replication_slots;
```

Для WAL retention и возможного удержания cleanup horizon.

## Частые бизнесовые симптомы и возможные причины

### "Диск растет, хотя мы удаляем данные"

Возможные причины:

- DELETE создает dead tuples;
- обычный VACUUM не возвращает место OS;
- autovacuum не успевает;
- long transactions мешают cleanup;
- replication slot держит WAL/старые версии;
- индексы раздулись;
- deleted данные не партиционированы.

Что делать:

- проверить `n_dead_tup`;
- проверить long transactions;
- проверить autovacuum;
- проверить размеры индексов;
- рассмотреть partitioning;
- для shrink - VACUUM FULL/pg_repack/CLUSTER/rewrite.

### "Запросы стали медленнее без изменения кода"

Возможные причины:

- table/index bloat;
- устарела статистика;
- visibility map деградировала;
- cache стал хуже из-за роста данных;
- autovacuum не успевает;
- новый индекс ухудшил запись;
- планы поменялись из-за распределения данных.

Что делать:

- `EXPLAIN (ANALYZE, BUFFERS)`;
- `pg_stat_statements`;
- размеры таблиц/индексов;
- `ANALYZE`;
- проверить autovacuum;
- проверить bloat/reindex candidates.

### "Реплика стала отставать"

Возможные причины:

- массовый UPDATE/DELETE;
- CREATE INDEX;
- много WAL;
- GIN/JSONB update-heavy workload;
- checkpoint pressure;
- slow replica IO;
- logical slot/consumer отстал.

Что делать:

- смотреть WAL generation;
- смотреть replication lag;
- батчить операции;
- избегать гигантских транзакций;
- планировать миграции.

### "Autovacuum постоянно грузит базу"

Возможные причины:

- таблица update-heavy;
- autovacuum долго игнорировали, теперь он догоняет;
- слишком низкие thresholds;
- много индексов;
- bloat;
- long transactions мешают завершить полезную очистку.

Что делать:

- не отключать вслепую;
- понять, какие таблицы;
- настроить per-table параметры;
- убрать лишние индексы;
- изменить write pattern;
- решить long transactions;
- возможно сделать manual vacuum/repack в окно обслуживания.

### "VACUUM идет, но dead tuples не уменьшаются"

Возможные причины:

- старые транзакции удерживают snapshot;
- prepared transactions;
- replication slots;
- VACUUM не может получить нужные locks;
- dead tuple estimate неточный;
- нагрузка создает мусор быстрее, чем VACUUM чистит.

Что делать:

- проверить `pg_stat_activity`;
- проверить `pg_replication_slots`;
- проверить prepared transactions;
- смотреть progress vacuum;
- проверить настройки autovacuum.

## Как ревьюить backend-код с учетом MVCC/VACUUM

### 1. Долгие транзакции

Плохо:

```go
tx, err := db.BeginTx(ctx, nil)
if err != nil {
	return err
}
defer tx.Rollback()

order, err := repo.GetOrderTx(ctx, tx, id)
if err != nil {
	return err
}

resp, err := paymentClient.Charge(ctx, order)
if err != nil {
	return err
}

err = repo.MarkPaidTx(ctx, tx, id, resp.PaymentID)
if err != nil {
	return err
}

return tx.Commit()
```

Проблема:

- транзакция открыта во время внешнего payment call;
- держит snapshot/locks дольше;
- может мешать VACUUM;
- может усиливать lock contention;
- при ретраях бизнес-эффект сложнее.

Лучше проектировать так, чтобы DB-транзакция была короткой.

### 2. Обновление часто меняемых полей

Плохо:

```go
_, err := db.ExecContext(ctx, `
	UPDATE users
	SET last_seen_at = now()
	WHERE id = $1
`, userID)
```

на каждый request.

Проверить:

- есть ли индекс по `last_seen_at`;
- насколько часто update;
- нужен ли real-time;
- можно ли throttle;
- можно ли вынести в Redis/events.

### 3. Массовые операции без батчей

Плохо:

```sql
DELETE FROM logs
WHERE created_at < now() - interval '90 days';
```

на 500 млн строк.

Лучше:

- partition drop;
- батчи;
- maintenance window;
- мониторинг WAL/replication;
- manual vacuum/analyze после.

### 4. Индекс на колонку, которая часто меняется

Проверить:

- убивает ли HOT;
- нужен ли индекс реально;
- можно ли partial index;
- можно ли другой порядок колонок;
- не лучше ли отдельная таблица.

### 5. Soft delete без partial indexes

Плохо:

```sql
SELECT *
FROM documents
WHERE account_id = $1
  AND deleted_at IS NULL
ORDER BY created_at DESC;
```

без индекса:

```sql
ON documents(account_id, created_at DESC)
WHERE deleted_at IS NULL
```

И еще проверить retention deleted rows.

### 6. Большой JSONB в горячей таблице

Проверить:

- часто ли обновляется JSON;
- есть ли GIN;
- можно ли нормализовать;
- можно ли append-only events;
- не растет ли TOAST и WAL.

### 7. SELECT в долгой транзакции

Даже read-only транзакция может вредить, если она долго держит snapshot.

Особенно:

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
-- большой отчет на primary
```

Для отчетов лучше:

- replica;
- statement_timeout;
- read-only transaction с контролем длительности;
- аналитическое хранилище;
- materialized views/aggregates.

## Методология диагностики bloat/VACUUM проблемы

Если видишь симптомы: диск растет, latency растет, autovacuum шумит, индексы огромные.

Порядок действий:

1. Найти самые большие relations.

   ```sql
   SELECT
   	relname,
   	pg_size_pretty(pg_total_relation_size(relid)) AS total_size
   FROM pg_catalog.pg_statio_user_tables
   ORDER BY pg_total_relation_size(relid) DESC
   LIMIT 20;
   ```

2. Сравнить table size и index size.

3. Посмотреть `n_live_tup`, `n_dead_tup`, `last_autovacuum`.

4. Проверить long transactions.

5. Проверить replication slots.

6. Проверить update-heavy таблицы и HOT ratio.

7. Проверить неиспользуемые/дублирующие индексы.

8. Посмотреть планы проблемных запросов.

9. Определить тип проблемы:

   - autovacuum не успевает;
   - cleanup horizon удерживается;
   - таблица нуждается в rewrite;
   - index bloat требует reindex;
   - write pattern надо менять;
   - нужен partitioning.

10. Выбрать безопасное действие.

    - tuning autovacuum;
    - manual vacuum;
    - analyze;
    - reindex concurrently;
    - pg_repack;
    - partition maintenance;
    - application fix.

## Типичные ошибки

### Ошибка 1: "DELETE освободит диск"

Нет. DELETE делает строки невидимыми для будущих транзакций, но физическое место обычно освобождается позже и чаще для переиспользования внутри таблицы, а не для OS.

### Ошибка 2: "VACUUM FULL - обычная регулярная операция"

Нет. Это тяжелый rewrite с блокировками. Регулярная гигиена - autovacuum и обычный VACUUM.

### Ошибка 3: "Autovacuum можно выключить, чтобы не мешал"

Очень опасно. Можно получить wraparound risk, bloat, плохие планы, деградацию.

### Ошибка 4: "Индекс на updated_at безвреден"

Если `updated_at` меняется при каждом update, индекс по нему может сильно увеличить write cost и убить HOT.

### Ошибка 5: "Долгая read-only транзакция безопасна"

Она может удерживать snapshot и мешать VACUUM удалять старые версии.

### Ошибка 6: "REINDEX решает все"

REINDEX помогает index bloat, но не исправляет:

- плохой SQL;
- table bloat;
- long transactions;
- слишком много UPDATE;
- лишние индексы;
- неверную модель данных.

### Ошибка 7: "Если VACUUM прошел, значит все очищено"

VACUUM мог не удалить recently dead tuples из-за старых snapshots. Нужно смотреть контекст.

### Ошибка 8: "Index Only Scan всегда не читает таблицу"

Index Only Scan зависит от visibility map. Если страницы не all-visible, будут heap fetches.

## Что отвечать на интервью

### Что такое MVCC?

MVCC - механизм многоверсионности. PostgreSQL при изменениях хранит несколько версий строки, а каждая транзакция читает данные согласно своему snapshot. Благодаря этому читатели и писатели меньше блокируют друг друга, но появляются старые версии строк, которые нужно очищать VACUUM.

### Что происходит при UPDATE?

PostgreSQL создает новую версию строки и помечает старую как замененную. Старая версия остается в heap, пока может быть видна старым транзакциям. Если HOT невозможен, для новой версии добавляются записи в индексы. Позже VACUUM очищает старые версии и индексные записи.

### Почему UPDATE дорогой?

Потому что это не просто изменение байтов на месте. Это новая tuple version, WAL, возможно новые index entries, dead tuple, будущая работа VACUUM. Если таблица имеет много индексов или меняется indexed column, цена растет.

### Что такое HOT update?

HOT update - оптимизация, когда PostgreSQL создает новую версию строки на той же heap-странице и не добавляет новые записи в индексы. Возможен, если изменяемые колонки не требуют обновления индексов и на странице есть место. HOT снижает index bloat и ускоряет update-heavy workload.

### Что делает VACUUM?

VACUUM удаляет dead tuples, очищает соответствующие index entries, обновляет free space map и visibility map, замораживает старые tuple при необходимости и помогает базе переиспользовать место. Обычный VACUUM обычно не возвращает место операционной системе.

### Чем VACUUM отличается от VACUUM FULL?

Обычный VACUUM чистит мусор и делает место reusable внутри relation. VACUUM FULL переписывает таблицу компактно и может вернуть место OS, но требует тяжелой блокировки и является maintenance-операцией.

### Почему место на диске не уменьшилось после DELETE и VACUUM?

DELETE создает dead tuples. VACUUM очищает их и делает место доступным для переиспользования внутри файла таблицы, но обычно не уменьшает сам файл. Чтобы реально вернуть место OS, нужен rewrite: VACUUM FULL, CLUSTER, pg_repack, пересоздание таблицы или drop partition.

### Что такое bloat?

Bloat - раздувание таблиц или индексов из-за dead tuples, page splits, старых версий строк и неидеального переиспользования места. Он увеличивает размер на диске, ухудшает cache locality, замедляет scans, backup, restore и обслуживание.

### Почему долгие транзакции опасны?

Они держат старый snapshot. Пока snapshot активен, VACUUM не может удалить версии строк, которые потенциально видимы этой транзакции. Это приводит к росту dead tuples, bloat и проблемам с autovacuum.

### Когда нужен REINDEX?

Когда индекс раздут, поврежден или его нужно физически перестроить. REINDEX помогает index bloat, но не лечит table bloat и не исправляет плохой write pattern. В продакшене часто используют REINDEX CONCURRENTLY, чтобы уменьшить блокировки.

### Как понять, что autovacuum не справляется?

Смотреть `pg_stat_user_tables`: `n_dead_tup`, `last_autovacuum`, `autovacuum_count`, `n_tup_upd`, `n_tup_hot_upd`. Проверить long transactions в `pg_stat_activity`, replication slots, progress vacuum, размеры таблиц и индексов.

## Мини-шпаргалка

`INSERT`:

- новая tuple;
- index entries во все индексы;
- WAL;
- dirty pages.

`UPDATE`:

- старая tuple получает `xmax`;
- новая tuple получает `xmin`;
- non-HOT update добавляет index entries;
- старая версия чистится VACUUM позже.

`DELETE`:

- tuple помечается удаленной;
- физически не исчезает сразу;
- index entries чистятся позже.

`HOT`:

- новая версия на той же странице;
- без новых index entries;
- нужен запас места и отсутствие изменений indexed columns.

`VACUUM`:

- чистит dead tuples;
- чистит indexes;
- обновляет FSM/VM;
- freeze;
- обычно не shrink disk file.

`VACUUM FULL`:

- rewrite;
- shrink;
- тяжелая блокировка.

`ANALYZE`:

- статистика для planner;
- не чистит dead tuples.

`REINDEX`:

- перестроить индекс;
- лечит index bloat;
- требует место/IO/locks.

`Bloat`:

- лишний физический размер;
- хуже cache;
- медленнее scans;
- больше backup/restore;
- часто следствие UPDATE/DELETE и плохого vacuum horizon.

## Хороший рассказ на 2 минуты

PostgreSQL использует MVCC: при изменении строки он обычно не перезаписывает ее на месте, а создает новую версию. У каждой версии есть служебная информация вроде `xmin` и `xmax`, а транзакция читает данные через snapshot. Это позволяет читателям и писателям меньше блокировать друг друга, но старые версии строк остаются в heap. Когда они уже не видны ни одной активной транзакции, VACUUM может их очистить, обновить free space map, visibility map и убрать мертвые ссылки из индексов. Обычный VACUUM обычно не возвращает место операционной системе, а делает его пригодным для повторного использования. Если таблица или индекс сильно раздулись, нужны более тяжелые операции: REINDEX для индексов, VACUUM FULL/CLUSTER/pg_repack или partition drop для таблиц. При UPDATE индексы обновляются, если HOT update невозможен; HOT возможен, когда изменяемые колонки не требуют изменения индексов и новая версия помещается на той же странице. Поэтому на практике опасны частые update indexed columns, long-running transactions, idle in transaction, массовые delete/update и слабый autovacuum. Для диагностики я бы смотрел `pg_stat_user_tables`, `pg_stat_activity`, `pg_stat_user_indexes`, размеры relations, `pg_stat_progress_vacuum`, replication slots и планы через `EXPLAIN (ANALYZE, BUFFERS)`.
