# Подсистемы хранения и извлечения данных

Эта глава DDIA не про конкретную базу данных и не про синтаксис запросов.
Главная мысль: база данных должна не только принять запись, но и потом быстро
найти нужные данные. Для этого storage engine выбирает физическую модель
хранения, а индекс становится производной структурой, которая ускоряет чтение
ценой дополнительной работы на запись.

Вся глава держится на сравнении двух больших подходов: log-structured storage,
где записи добавляются в конец файлов и затем сливаются, и page-oriented
storage, где данные организованы в страницы, которые можно читать и изменять
на месте.

## 1. Индекс как производная структура

Индекс - это дополнительная структура данных, построенная поверх основных
данных. Он помогает быстрее находить записи, но не является бесплатным:
каждая запись в базу теперь должна обновлять не только сами данные, но и
индексы.

Простейшая модель key-value storage: есть файл-журнал, куда записи только
добавляются в конец, и есть индекс, который говорит, где в файле лежит последнее
значение ключа.

```text
key -> byte offset in log
```

Такой индекс может быть обычной hash table в памяти. При записи нового значения
оно добавляется в конец файла, а hash table обновляет смещение для ключа.
Актуальным считается последнее значение.

Сила подхода - простая и быстрая запись. Слабость - индекс должен помещаться в
оперативную память, а поиск по диапазонам ключей неудобен: hash table хорошо
отвечает на точечный запрос по ключу, но не хранит ключи в порядке.

## 2. Журнал, сегменты и compaction

Append-only журнал не может расти бесконечно одним файлом. Его разбивают на
сегменты. Старые сегменты можно уплотнять: если один ключ встречался несколько
раз, оставляется только последняя версия.

```text
before:
  a = 1
  b = 2
  a = 3

after compaction:
  b = 2
  a = 3
```

После уплотнения сегменты можно сливать. Это позволяет постепенно освобождать
место, удалять старые версии и поддерживать manageable набор файлов.

Удаление в такой модели обычно записывается не как физическое стирание из
середины файла, а как tombstone - специальная запись "этот ключ удален". Позже
во время compaction старые значения ключа и сам tombstone могут быть выброшены,
когда они больше не нужны.

Для надежности у журнала есть практические детали: бинарный формат записей,
длины записей, checksums для обнаружения недописанных данных, восстановление
индекса после падения через перечитывание сегментов или snapshot индекса.

## 3. SSTables

SSTable - sorted string table: сегмент, в котором пары key-value отсортированы
по ключу. Это важный шаг от простого append-only журнала.

Сортировка дает несколько преимуществ.

Во-первых, несколько SSTable можно эффективно сливать как отсортированные
последовательности. Если один и тот же ключ встречается в нескольких сегментах,
берется самая новая версия.

Во-вторых, индекс может быть sparse. Не нужно хранить offset для каждого ключа:
достаточно хранить смещения для некоторых ключей и читать небольшой блок между
ними.

```text
index:
  apple  -> block 1
  car    -> block 2
  dog    -> block 3
```

Если нужен ключ `cat`, можно найти ближайшую известную позицию и просканировать
не весь файл, а небольшой диапазон. На практике данные группируют в блоки и
сжимают, а индекс указывает на начало блока.

## 4. Memtable и LSM-tree

Чтобы создавать SSTable, новые записи сначала собирают в памяти в структуре,
которая поддерживает ключи в отсортированном порядке. Такую структуру называют
memtable.

Путь записи выглядит так:

```text
write
  -> append to recovery log
  -> update memtable
  -> flush memtable as SSTable
  -> background merge/compaction
```

Recovery log нужен для восстановления memtable после сбоя. Он не обязан быть
удобным для чтения: его задача - durability до сброса memtable на диск.

LSM-tree - log-structured merge-tree - строится вокруг этой идеи: данные
сначала пишутся в память, затем сбрасываются в отсортированные immutable
файлы, а потом фоновые процессы сливают эти файлы в более крупные уровни или
сегменты.

Сильная сторона LSM - запись. Storage engine превращает множество изменений в
последовательные записи и фоновые merge-операции. Цена - чтение может смотреть
в несколько мест: memtable и несколько SSTable. Bloom filters помогают быстро
пропускать файлы, где ключа точно нет.

## 5. B-trees

B-tree представляет другой подход. Вместо append-only сегментов база делит
хранилище на страницы фиксированного размера и организует из них сбалансированное
дерево.

Поиск идет от root page к leaf page. Внутренние страницы содержат ключи-разделители
и ссылки на дочерние страницы. В leaf pages находятся сами ключи или ссылки на
строки.

Ключевая идея - fanout: в одной странице помещается много ключей и ссылок на
следующие страницы. Поэтому высота B-tree мала. В отличие от бинарного дерева,
где ветвление равно двум, B-tree использует большой коэффициент ветвления,
связанный с размером страницы.

Запись в B-tree меняет страницы на месте. Если страница переполнена, она
делится на две, а в родительскую страницу добавляется новый разделитель.

Так как изменение дерева должно быть надежным, перед изменением страниц обычно
используется write-ahead log: сначала изменения записываются в журнал
восстановления, затем применяются к страницам.

## 6. LSM и B-tree

LSM и B-tree решают одну задачу разными способами.

LSM делает запись дешевой в моменте: новые данные быстро попадают в память и
append-only log, затем сбрасываются в SSTable. Но позже система платит цену
compaction: нужно сливать файлы, удалять старые версии, поддерживать уровни.

B-tree чаще платит цену сразу: надо найти страницу, изменить ее, иногда
сделать split, записать WAL. Зато чтение по ключу и диапазону обычно идет по
предсказуемому пути через дерево.

Упрощенная память:

- LSM хорошо подходит для write-heavy нагрузки и последовательной записи;
- B-tree хорошо подходит для предсказуемых точечных и диапазонных чтений;
- LSM может иметь несколько копий одного ключа до compaction;
- в B-tree ключ в индексе обычно находится в одном месте;
- LSM часто лучше сжимается;
- B-tree может страдать от фрагментации страниц.

## 7. Другие индексы

Первичный индекс определяет основной путь доступа к данным по primary key.
Вторичные индексы позволяют искать те же данные по другим полям.

Индекс может хранить ссылку на строку в heap/table storage. В clustered index
данные физически хранятся рядом с ключом индекса или в порядке индекса,
в зависимости от конкретной системы.

Компромисс между этими вариантами - covering index: индекс хранит не только
ключи поиска, но и дополнительные колонки, нужные запросу. Если запрос можно
ответить только из индекса, не нужно идти за строкой в основное хранилище.

Составной индекс хранит несколько колонок вместе и особенно полезен, когда
запросы используют их в том порядке, в котором индекс построен.

Отдельная категория - полнотекстовые и fuzzy indexes. Они нужны не для точного
поиска ключа, а для поиска слов, близких написаний, синонимов или фрагментов
текста.

## 8. OLTP и OLAP

Вторая часть главы переключает внимание с storage engine для транзакционных
запросов на аналитические системы.

OLTP - это обычная прикладная нагрузка: много коротких запросов, чтение или
обновление небольшого числа строк, работа с пользователями, заказами, платежами,
профилями.

OLAP - аналитическая нагрузка: запросы читают много строк, часто выбирают
несколько колонок и считают агрегаты. Это отчеты, статистика, BI, продуктовая
аналитика.

Поэтому операционные базы и аналитические хранилища часто разделяют. Данные из
OLTP-систем извлекаются, преобразуются и загружаются в data warehouse, где
структуры хранения оптимизированы под большие сканы и агрегации.

## 9. Data warehouse и column storage

В data warehouse часто используется star schema или snowflake schema. В центре
находится fact table с событиями или фактами бизнеса, вокруг - dimension tables:
кто, где, когда, какой продукт, какой канал.

Для аналитики row-oriented storage часто читает слишком много лишних данных.
Если запросу нужны только `city`, `date` и `amount`, не хочется читать всю
широкую строку целиком.

Column storage хранит значения каждой колонки отдельно. Позиция значения в
одной колонке соответствует той же логической строке в других колонках.

```text
city:    msk, spb, msk, ...
amount:  100, 200, 150, ...
date:    d1,  d1,  d2,  ...
```

Это позволяет читать только нужные колонки и хорошо сжимать данные. Сжатие
особенно эффективно, когда в колонке много повторяющихся значений или данные
удачно отсортированы.

В главе важны несколько идей:

- bitmap encoding хорошо работает для колонок с ограниченным числом разных
  значений;
- run-length encoding выигрывает, когда одинаковые значения идут длинными
  сериями;
- сортировка строк помогает сжатию, но нужно сохранять одинаковый порядок во
  всех колонках;
- vectorized execution обрабатывает данные пачками и лучше использует CPU.

Цена column storage - более сложные записи и обновления. Эти приемы особенно
сильны там, где данные читаются большими аналитическими запросами.

## 10. Materialized views и cubes

Materialized view - сохраненный результат запроса. Он ускоряет чтение, потому
что часть работы уже выполнена заранее, но добавляет задачу обновления этого
результата при изменении исходных данных.

Data cube - похожая идея для агрегатов по нескольким измерениям: дата, город,
категория товара, канал продаж. Вместо пересчета всего отчета с нуля система
может хранить заранее подготовленные агрегаты.

## Как выбирать подсистему хранения

Короткая эвристика главы:

- если важны частые точечные записи и высокий write throughput, вспоминаем
  log-structured подходы, SSTables и LSM;
- если важны предсказуемые чтения по ключу и диапазону, вспоминаем B-tree;
- если ключей слишком много, hash index в памяти становится проблемой;
- если нагрузка аналитическая и читает несколько колонок из огромных таблиц,
  вспоминаем column storage;
- если отчет часто повторяется, можно подумать о materialized view или cube;
- если создаем индекс, надо помнить, что он ускоряет чтение ценой записи и
  места.

## Ответ на собеседовании

Storage engine - это способ, которым база физически хранит и извлекает данные.
Индекс является производной структурой: он ускоряет чтение, но добавляет цену
на запись.

В log-structured подходе записи добавляются в конец журнала. Чтобы журнал не
рос бесконечно, его делят на сегменты, уплотняют и сливают. SSTable делает
сегменты отсортированными, memtable собирает новые записи в памяти, а LSM-tree
организует каскадное слияние таких файлов.

B-tree использует страницы фиксированного размера и сбалансированное дерево.
За счет большого fanout высота дерева мала, поэтому чтение по ключу и диапазону
предсказуемое. Цена - обновление страниц, split и WAL.

Для OLTP обычно важны короткие транзакционные запросы. Для OLAP важны большие
сканы, агрегации и чтение отдельных колонок, поэтому в warehouse часто
используют column storage, сжатие, vectorized execution, materialized views и
cubes.
