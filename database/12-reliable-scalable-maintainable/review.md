# Надежные, масштабируемые и сопровождаемые приложения

Первая глава DDIA задает словарь для всей книги. Она не про конкретную базу,
алгоритм репликации или формат хранения. Она про то, какими свойствами должна
обладать система данных, чтобы с ней можно было жить: она должна продолжать
работать при сбоях, выдерживать рост нагрузки и оставаться понятной для людей,
которые ее развивают.

Три главных свойства:

- reliability - система продолжает работать правильно, даже когда что-то идет
  не так;
- scalability - система сохраняет приемлемую производительность при росте
  нагрузки;
- maintainability - систему можно понимать, эксплуатировать и менять со
  временем.

Главная мысль главы: эти свойства нельзя "добавить потом" одной технологией.
Они возникают из инженерных решений, компромиссов и операционной дисциплины.

## 1. Data-intensive applications

Книга говорит о data-intensive applications: приложениях, где главная сложность
не в CPU-вычислениях, а в данных.

Такие системы обычно комбинируют несколько компонентов:

- database - хранить данные и находить их позже;
- cache - ускорять чтения или снижать нагрузку на источник истины;
- search index - искать по тексту или сложным признакам;
- stream processing - обрабатывать события асинхронно;
- batch processing - пересчитывать большие объемы данных.

Для приложения это не набор независимых коробок. Вместе они образуют одну
систему данных:

```text
request
  -> service
       -> database
       -> cache
       -> search index
       -> message queue
       -> background workers
```

Поэтому проектирование системы данных - это не только выбор PostgreSQL, Redis
или Kafka. Нужно понимать, где источник истины, какие данные могут отставать,
как восстанавливаются индексы, как переживаются сбои и что видит пользователь.

## 2. Reliability: что значит "работает правильно"

Reliability - это способность системы продолжать выполнять свою функцию даже
при неблагоприятных обстоятельствах.

Важно различать fault и failure:

- fault - одна часть системы работает неправильно: диск умер, сеть отвалилась,
  процесс упал, запрос вернул ошибку;
- failure - система как целое перестала выполнять обещанную пользователю
  функцию.

Надежная система не обещает, что fault никогда не случится. Наоборот, она
ожидает faults и старается не превращать их в user-visible failure.

Пример:

```text
fault:
  один follower database отстал или упал

не обязательно failure:
  leader продолжает принимать записи, чтения переключаются на другие replicas
```

Инженерная цель - уменьшать вероятность failures и ограничивать последствия
faults.

## 3. Hardware faults

Аппаратные сбои обычны: диски ломаются, память дает ошибки, питание пропадает,
машины перезагружаются. Исторически часть надежности достигалась через более
дорогое железо и избыточность внутри одной машины: RAID, двойные блоки питания,
hot-swappable components.

Но в больших системах отказ отдельной машины становится нормальным событием.
Поэтому подход смещается от "сделать одну машину почти неубиваемой" к "сделать
систему, которая переживает потерю машин".

Практический вывод:

- хранить несколько копий данных;
- уметь переключаться на другую машину;
- не делать один узел единственной точкой отказа;
- регулярно проверять восстановление, а не только наличие backup.

## 4. Software errors

Software faults часто опаснее hardware faults, потому что они могут быть
коррелированными. Один bug может одновременно затронуть все узлы:

- редкий input вызывает crash во всех instances;
- неправильный config ломает весь кластер;
- memory leak постепенно съедает память на всех worker-ах;
- dependency начинает отдавать неожиданные ответы, и весь сервис входит в
  ошибочное состояние.

Такие сбои сложнее, потому что простое "добавим еще одну копию" не спасает,
если все копии выполняют один и тот же неправильный код.

Что помогает:

- тесты на edge cases;
- изоляция компонентов;
- контроль инвариантов;
- быстрый rollback;
- monitoring и alerting;
- defense in depth: проверки на нескольких уровнях;
- chaos/ fault injection там, где это оправдано.

## 5. Human errors

Люди тоже часть системы. Операционные ошибки, неверные команды, неудачные
миграции, плохой rollout и непонятные интерфейсы часто становятся причиной
инцидентов.

Глава подчеркивает не "убрать людей", а проектировать системы так, чтобы людям
было сложнее ошибиться и проще восстановиться.

Практические меры:

- понятные admin-инструменты;
- staging и production parity;
- безопасные default values;
- dry-run для опасных операций;
- gradual rollout и canary;
- хорошая observability;
- простой rollback;
- postmortems без поиска виноватого.

Надежность - это не только код. Это еще процессы эксплуатации.

## 6. Scalability: рост нагрузки без потери качества

Scalability не означает "система быстрая". Она означает: если нагрузка растет,
есть понятный способ сохранить приемлемую производительность.

Сначала нужно описать нагрузку. Нельзя обсуждать масштабирование абстрактно.
Нужны load parameters:

- requests per second;
- reads/writes ratio;
- размер данных;
- количество активных пользователей;
- fan-out на один запрос;
- размер payload;
- cardinality ключей;
- hot keys;
- распределение по регионам;
- количество background jobs.

Одна и та же архитектура может быть отличной при одном профиле нагрузки и
плохой при другом.

## 7. Пример с fan-out

Книга разбирает идею на примере социальной ленты. Есть два крайних подхода.

Fan-out on read:

```text
user opens home timeline
  -> find followed users
  -> fetch recent posts from all of them
  -> merge and sort
```

Плюс: запись поста дешевая. Минус: чтение ленты дорогое, особенно если
пользователь подписан на многих.

Fan-out on write:

```text
user writes post
  -> put post into timelines of all followers

follower opens timeline
  -> read precomputed timeline
```

Плюс: чтение быстрое. Минус: запись дорогая, особенно если у автора миллионы
подписчиков.

Вывод главы не в том, что один подход всегда лучше. Выбор зависит от workload.
Иногда нужен гибрид: обычным пользователям fan-out on write, а celebrities
обрабатываются иначе.

## 8. Performance: latency, response time, throughput

Performance нужно измерять аккуратно.

Response time - сколько времени клиент ждет ответ. Это не только время работы
сервера: сюда входят сеть, очередь перед обработкой, scheduling, downstream
dependencies.

Latency часто используют как время ожидания до начала/завершения обслуживания,
но в разговоре эти слова нередко смешивают. Для практики важнее не спорить о
термине, а явно сказать, что именно измеряется.

Throughput - сколько работы система выполняет за единицу времени:

```text
requests/sec
events/sec
MB/sec
rows/sec
```

Высокий throughput не гарантирует низкий response time. Система может
обрабатывать много запросов, но держать часть пользователей в длинной очереди.

## 9. Percentiles вместо среднего

Среднее значение скрывает хвосты. Если большинство запросов быстрые, но 1%
очень медленные, average может выглядеть нормально, а пользователи все равно
будут страдать.

Поэтому для response time используют percentiles:

- p50 - половина запросов быстрее этого значения;
- p95 - 95% запросов быстрее;
- p99 - 99% запросов быстрее;
- p999 - хвост для очень чувствительных систем.

Смысл:

```text
p50 показывает типичный опыт
p95/p99 показывают, что происходит с медленными запросами
```

Tail latency важна, потому что один пользователь делает много запросов, а один
экран может зависеть от нескольких backend calls. Если хотя бы один из них
попал в хвост, весь пользовательский сценарий стал медленным.

## 10. Подходы к масштабированию

Есть два грубых направления:

- vertical scaling - более мощная машина;
- horizontal scaling - больше машин.

Vertical scaling проще операционно, но имеет предел и может быть дорогим.
Horizontal scaling дает больше пространства для роста, но требует думать о
распределении данных, репликации, partitioning, coordination и отказах.

DDIA не продает "распределенность" как самоцель. Распределенная система почти
всегда сложнее. Ее выбирают, когда требования по нагрузке, отказоустойчивости
или географии оправдывают эту сложность.

## 11. Maintainability: систему будут менять люди

Maintainability - это способность системы оставаться удобной для эксплуатации
и развития.

Книга выделяет три аспекта:

- operability;
- simplicity;
- evolvability.

Это не второстепенные качества. Большая часть стоимости системы возникает после
первого запуска: инциденты, миграции, новые требования, onboarding инженеров,
изменение нагрузки, обновление зависимостей.

## 12. Operability

Operability означает: команде понятно, как система работает в проде и что с ней
делать при проблемах.

Хорошая operability включает:

- metrics, logs, traces;
- понятные dashboards;
- health checks;
- backup/restore procedures;
- visibility в очереди, lag, saturation, error rate;
- простые runbooks;
- возможность безопасно менять config;
- понятные failure modes.

Плохая система может быть "корректной" на бумаге, но ужасной в эксплуатации:
непонятно, зависла она или медленно работает; непонятно, где bottleneck;
невозможно восстановить данные; любое изменение требует ручной магии.

## 13. Simplicity

Simplicity в DDIA - не примитивность. Это уменьшение accidental complexity:
сложности, которая не следует из самой предметной области, а появляется из-за
случайной запутанности реализации.

Помогают:

- хорошие абстракции;
- явные границы компонентов;
- понятная модель данных;
- отсутствие лишних special cases;
- простые инварианты;
- удаление мертвого кода;
- предсказуемые operational procedures.

Сложность опасна тем, что ее трудно держать в голове. Когда никто не понимает
систему целиком, изменения становятся медленными и рискованными.

## 14. Evolvability

Evolvability - способность системы меняться под новые требования.

Требования почти всегда меняются:

- продукт находит новую аудиторию;
- нагрузка растет не так, как ожидалось;
- появляются новые интеграции;
- схема данных эволюционирует;
- старые предположения перестают быть верными.

Хорошая система не обязана заранее идеально знать будущее. Но она должна
оставлять пространство для изменений: совместимость форматов, миграции,
разделение ответственности, тестируемость, наблюдаемость и отказ от ненужной
жесткой связности.

## 15. Как глава связывает всю книгу

Первая глава задает рамку для следующих:

- storage engines влияют на performance и operational behavior;
- replication влияет на reliability и latency;
- partitioning влияет на scalability и complexity;
- transactions помогают correctness, но имеют цену;
- distributed systems добавляют новые failure modes;
- encoding и schema evolution нужны для evolvability.

То есть DDIA дальше разбирает не набор технологий, а инженерные компромиссы
между надежностью, масштабированием и сопровождаемостью.

## Ответ на собеседовании

Надежность, масштабируемость и сопровождаемость - это три базовые оси системы
данных. Reliability означает, что отдельные faults не превращаются в отказ всей
системы. Scalability начинается с описания load parameters и измерения response
time/throughput через percentiles, а не со слова "шардирование". Maintainability
означает, что систему можно эксплуатировать, понимать и менять: operability,
simplicity и evolvability. Хороший дизайн начинается не с выбора модной базы, а
с понимания workload, failure modes и будущей цены изменений.
