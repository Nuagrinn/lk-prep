# Многопоточность, общий доступ к памяти и data race в Go

Этот конспект про первый класс проблем из ревью: многопоточность, гонки данных, конкурентный доступ к `map`, непоследовательное использование mutex и общую методику поиска таких ошибок на код-ревью.

Главная идея простая: если несколько горутин могут одновременно читать и изменять одни и те же данные, у этих данных должен быть понятный владелец или понятная синхронизация. Если этого нет, программа может работать “почти всегда”, а потом падать, портить данные или вести себя по-разному при одинаковом входе.

## Простое объяснение

В Go горутины выполняются конкурентно. Это значит, что две части программы могут выполняться “рядом” во времени: одна горутина обрабатывает HTTP-запрос, другая читает из очереди, третья обновляет кеш, четвертая публикует события. Когда каждая горутина работает только со своими локальными переменными, проблем почти нет. Локальная переменная живет внутри конкретного вызова функции, и другая горутина обычно не может ее случайно испортить.

Проблемы начинаются там, где появляется общее состояние. Например, поле структуры:

```go
type Service struct {
	cache map[string]Order
}
```

Если один запрос читает `cache`, второй запрос пишет в `cache`, а третий параллельно перестраивает `cache`, то это уже общий ресурс. Сам факт, что поле находится внутри `Service`, не делает его потокобезопасным. Если один экземпляр `Service` используется многими запросами, его поля становятся общим состоянием для всех этих запросов.

В Go обычная `map` не является безопасной для конкурентного доступа. Особенно опасна запись. Если две горутины одновременно пишут в одну map или одна пишет, а другая читает, это может привести к data race, повреждению внутренней структуры map или runtime panic вида:

```text
fatal error: concurrent map writes
```

Иногда panic не будет. Это даже хуже, потому что баг становится тихим. Программа может вернуть устаревшие данные, потерять обновление, записать неконсистентное значение или вести себя нестабильно под нагрузкой.

Data race означает, что две горутины обращаются к одной и той же памяти одновременно, хотя бы одно обращение является записью, и между ними нет синхронизации. Важно: data race не обязательно означает panic. Это означает, что результат программы больше не определен нормальной логикой кода.

## Как это выглядело в нашем примере

В файле `review_task.go` есть поле:

```go
cache map[string]Order
mu    sync.RWMutex
```

Наличие `mu` показывает намерение автора: кеш нужно защищать mutex. В `GetOrder` это действительно делается:

```go
s.mu.RLock()
order, ok := s.cache[id]
s.mu.RUnlock()

s.mu.Lock()
s.cache[id] = order
s.mu.Unlock()
```

Но в других местах запись идет напрямую:

```go
s.cache[order.ID] = order
```

Это плохой признак на ревью. Если одно и то же поле иногда используется под mutex, а иногда без mutex, почти всегда это ошибка. Такой код говорит: “у нас есть правило синхронизации, но оно применяется не везде”.

Самое опасное здесь не то, что забыли пару строк `Lock/Unlock`. Самое опасное, что у поля нет жесткого контракта доступа. Любой новый метод может снова обратиться к `s.cache` напрямую. Чем больше кодовая база, тем чаще такие ошибки появляются повторно.

## Почему mutex нужен и что он гарантирует

Mutex в Go решает две задачи.

Первая задача очевидная: он не дает двум горутинам одновременно выполнять критическую секцию. Если одна горутина вошла в блок между `Lock()` и `Unlock()`, другая будет ждать.

Вторая задача менее очевидная, но важная: mutex создает отношение синхронизации между горутинами. Грубо говоря, изменения, сделанные до `Unlock()`, становятся корректно видимыми для другой горутины после `Lock()`. Без такой синхронизации нельзя надежно рассуждать о видимости изменений между CPU core, кешами процессора и оптимизациями компилятора.

`sync.RWMutex` дает два режима. `RLock` разрешает нескольким читателям одновременно читать данные. `Lock` дает эксклюзивный доступ для записи. Для map это типичный паттерн: много чтений, меньше записей.

Но `RWMutex` полезен только если правило соблюдается везде. Один прямой доступ к map без mutex ломает всю модель.

## Правильный базовый паттерн

Самый простой вариант: каждое обращение к shared map защищено mutex.

```go
func (s *Service) getCachedOrder(id string) (Order, bool) {
	s.mu.RLock()
	defer s.mu.RUnlock()

	order, ok := s.cache[id]
	return order, ok
}

func (s *Service) setCachedOrder(order Order) {
	s.mu.Lock()
	defer s.mu.Unlock()

	s.cache[order.ID] = order
}
```

После этого остальные методы сервиса не должны писать `s.cache[...]` напрямую. Они вызывают `getCachedOrder` и `setCachedOrder`. Это маленькая инкапсуляция, но она сильно снижает шанс повторной ошибки.

Хорошее ревью-замечание здесь звучит так:

> Я бы спрятал прямой доступ к `cache` за методы, потому что сейчас синхронизация зависит от дисциплины каждого места вызова. Уже есть места, где mutex обошли.

Это сильнее, чем просто “добавь lock вот тут”. Ты показываешь не только локальный баг, но и причину, по которой баг легко повторится.

## Когда лучше заменить всю map одним действием

В `RebuildCache` часто возникает желание идти по заказам и писать их в кеш по одному:

```go
for _, order := range orders {
	s.mu.Lock()
	s.cache[order.ID] = order
	s.mu.Unlock()
}
```

Так уже безопаснее, чем писать без mutex, но не всегда идеально. Если мы перестраиваем кеш целиком, лучше собрать новую map локально, а затем под lock заменить ссылку:

```go
next := make(map[string]Order, len(orders))
for _, order := range orders {
	next[order.ID] = order
}

s.mu.Lock()
s.cache = next
s.mu.Unlock()
```

Локальная map `next` не является shared state, пока она не присвоена в `s.cache`. Поэтому ее можно спокойно заполнять без mutex. Lock держится очень короткое время: только на момент замены. Это уменьшает блокировки для читателей.

Но тут важно понимать бизнес-смысл. Если кеш общий для всех пользователей, а `RebuildCache(ctx, userID)` перестраивает кеш только для одного пользователя, полная замена `s.cache = next` может удалить заказы других пользователей. Значит, правильное решение зависит от модели данных. На ревью это можно сформулировать как вопрос:

> Этот кеш глобальный или пользовательский? Если метод перестраивает данные одного пользователя, не потеряем ли мы кеш других пользователей?

Это хороший пример, когда concurrency-проблема пересекается с архитектурой данных.

## Не всякая синхронизация обязана быть mutex

Mutex не единственный способ сделать конкурентный код безопасным. Есть несколько нормальных моделей.

Первая модель: shared state плюс mutex. Это подходит для простых структур вроде кеша, счетчиков, таблиц состояния.

Вторая модель: владение через одну горутину. Например, только worker владеет map, а остальные горутины отправляют ему команды через channel. Тогда map вообще не требует mutex, потому что физически к ней обращается только одна горутина.

Третья модель: immutable data. Мы не меняем существующую структуру, а создаем новую и атомарно заменяем ссылку. Для некоторых read-heavy сценариев можно использовать `atomic.Value`.

Четвертая модель: готовые concurrent-примитивы. Например, `sync.Map`, но его не стоит использовать как универсальную замену обычной map. `sync.Map` хорош для специальных сценариев: много конкурентных чтений, редкие записи, ключи живут долго, или ключи распределены между независимыми горутинами. Для обычного сервисного кеша `map + RWMutex` часто проще и понятнее.

На код-ревью не нужно автоматически требовать `sync.Map`. Чаще правильнее спросить: “какая модель владения данными здесь выбрана?” Если ответ не виден из кода, это уже проблема дизайна.

## Типичные признаки data race на ревью

Самый частый признак: поле структуры меняется из методов, которые могут вызываться параллельно. Например, сервис создается один раз и используется HTTP-хендлерами. Значит, почти любой метод сервиса может быть вызван одновременно с другим методом сервиса.

Второй признак: есть `go func()` и внутри используется переменная или поле из внешней области. Нужно проверить, что именно захватывает горутина, как долго она живет и кто еще работает с этими данными.

Третий признак: есть `map`, `slice`, указатель на структуру, счетчик, кеш, буфер, очередь, состояние подключения, last seen timestamp, in-memory registry. Все это потенциально shared mutable state.

Четвертый признак: mutex есть, но используется не рядом со всеми обращениями к данным. Особенно подозрительно, когда чтение под lock, а запись без lock, или наоборот.

Пятый признак: метод называется как read-only, но меняет состояние. Например, `ListUserOrders` только перечисляет заказы, но параллельно запускает goroutine и публикует событие. Это уже не чистое чтение. У таких методов выше шанс неожиданных гонок и сайд-эффектов.

Шестой признак: в коде есть фоновые worker'ы, `Close`, `Start`, `Stop`, каналы и кеши. Жизненный цикл фоновых горутин почти всегда связан с конкурентным доступом.

## Методика диагностики на код-ревью

На ревью удобно идти не хаотично, а по маршруту. Тогда ты не зависишь от вдохновения и быстрее находишь проблемы.

Сначала нужно понять, какие объекты долгоживущие. В сервисном слое это обычно структура `Service`, репозитории, клиенты внешних сервисов, кеши, каналы, worker'ы. Если объект создается один раз и живет все время работы приложения, его поля могут использоваться многими горутинами.

Затем нужно найти mutable state. Mutable state - это все, что может изменяться после создания: map, slice, поля структур, счетчики, каналы, указатели на изменяемые объекты. В нашем примере это `cache`, `jobs`, `stop`, частично `metrics` и внешние клиенты, если они не потокобезопасны.

После этого для каждого mutable field нужно задать вопрос: кто владелец этих данных? Есть ли один владелец или доступ распределен между многими методами? Если доступ распределен, какая синхронизация используется?

Дальше полезно выписать все места чтения и записи. Для `cache` это:

```text
GetOrder: read under RLock
GetOrder: write under Lock
CreateOrder: write without Lock
RebuildCache: write without Lock
```

Такой список сразу показывает проблему. На собеседовании необязательно реально выписывать его на бумаге, но мыслить стоит именно так.

Потом нужно проверить, не утекает ли ссылка на shared data наружу. Например, если метод возвращает внутренний slice или map, вызывающий код может изменить его без lock. В нашем примере `Order.Items` - это slice. Если `Order` хранится в кеше и возвращается наружу как значение, сам заголовок slice копируется, но underlying array остается общим. Если кто-то снаружи изменит `order.Items[0]`, это может неожиданно изменить данные, которые логически считались кешированными. Для middle-секции это уже более продвинутое замечание, но оно хорошее.

После этого нужно посмотреть на горутины. Каждая `go func()` задает вопросы: чем она владеет, какие переменные захватывает, кто ждет ее завершения, что будет при отмене контекста, куда деваются ошибки, может ли горутин стать слишком много.

И наконец нужно посмотреть на lifecycle. Если есть `Close`, `Stop`, `Start`, worker или канал, нужно проверить: можно ли вызвать `Close` дважды, что происходит с уже отправленными задачами, может ли отправитель писать в канал после остановки, есть ли `WaitGroup`, есть ли контекст остановки.

## Как формулировать замечания на ревью

Сильное ревью-замечание обычно состоит из четырех частей: место, проблема, последствие, предложение.

Плохо:

```text
Тут race.
```

Лучше:

```text
В CreateOrder запись в s.cache идет без s.mu.Lock, хотя GetOrder читает этот же cache под RLock. Если CreateOrder и GetOrder выполнятся параллельно, будет data race и возможна panic concurrent map read/write. Я бы вынес доступ к cache в методы get/set или везде строго использовал один mutex.
```

Такое замечание хорошо звучит на собеседовании, потому что оно показывает причинно-следственную цепочку. Ты не просто называешь термин, а объясняешь, что именно может случиться.

Для архитектурного замечания:

```text
Сейчас cache - shared mutable state внутри долгоживущего Service, но контракт доступа к нему не закреплен. Часть кода использует mutex, часть обращается напрямую. Я бы запретил прямой доступ к map за пределами маленьких helper-методов или выделил отдельный cache-объект с собственными методами.
```

Это уже не просто поиск ошибки, а предложение по снижению риска класса ошибок.

## Как можно дописать наш пример

Минимальная доработка выглядит так:

```go
func (s *Service) getCachedOrder(id string) (Order, bool) {
	s.mu.RLock()
	defer s.mu.RUnlock()

	order, ok := s.cache[id]
	return order, ok
}

func (s *Service) setCachedOrder(order Order) {
	s.mu.Lock()
	defer s.mu.Unlock()

	s.cache[order.ID] = order
}
```

Тогда `GetOrder` становится проще:

```go
func (s *Service) GetOrder(ctx context.Context, id string) (Order, error) {
	if order, ok := s.getCachedOrder(id); ok {
		return order, nil
	}

	order, err := s.repo.GetOrder(ctx, id)
	if err != nil {
		return Order{}, err
	}

	s.setCachedOrder(order)
	return order, nil
}
```

В `CreateOrder` после успешного commit:

```go
s.setCachedOrder(order)
```

В `RebuildCache`, если нужно обновить только заказы пользователя, можно сделать отдельный метод. Но для этого в `Order` должен быть `UserID`, и нужно аккуратно удалить старые заказы этого пользователя, сохранив чужие:

```go
func (s *Service) replaceUserCache(userID string, orders []Order) {
	s.mu.Lock()
	defer s.mu.Unlock()

	for id, order := range s.cache {
		if order.UserID == userID {
			delete(s.cache, id)
		}
	}

	for _, order := range orders {
		s.cache[order.ID] = order
	}
}
```

Это безопасно с точки зрения race, но может быть не лучшим решением по производительности, если кеш большой. Тогда уже нужно думать о структуре `map[userID]map[orderID]Order` или о полноценном cache-компоненте.

## Про defer Unlock

В простых методах хорошо использовать `defer Unlock()` сразу после `Lock()`. Это снижает шанс забыть unlock при раннем return.

```go
s.mu.Lock()
defer s.mu.Unlock()
```

Но не стоит держать lock во время медленных операций: запросов в БД, сетевых вызовов, публикации событий, ожидания channel, тяжелых вычислений. Lock должен защищать только доступ к shared state. Например, так плохо:

```go
s.mu.Lock()
order, err := s.repo.GetOrder(ctx, id)
s.cache[id] = order
s.mu.Unlock()
```

Здесь mutex удерживается во время запроса в БД. Все другие читатели кеша будут ждать БД, хотя сам кеш в этот момент не используется. Лучше:

```go
order, err := s.repo.GetOrder(ctx, id)
if err != nil {
	return Order{}, err
}

s.mu.Lock()
s.cache[id] = order
s.mu.Unlock()
```

На ревью стоит обращать внимание не только на наличие mutex, но и на размер критической секции. Слишком большая критическая секция может привести к деградации производительности или даже deadlock, если внутри вызывается чужой код.

## Не вызывать внешний код под lock

Хорошее правило: под lock не стоит вызывать неизвестный внешний код. Например, не надо держать mutex и внутри вызывать `events.Publish`, `repo.GetOrder`, `payments.Charge`, пользовательский callback или интерфейсный метод, если не понятно, что он делает.

Причина простая: внешний код может быть медленным, может попытаться взять другой lock, может вызвать обратно этот же сервис, может зависнуть. Тогда маленькая защита map превращается в глобальную блокировку сервиса.

В нашем примере прямого вызова внешнего кода под `mu` нет, и это хорошо. Ошибка именно в обратную сторону: mutex не используется там, где должен.

## Race detector

В Go есть встроенный race detector:

```bash
go test -race ./...
```

Он запускает тесты с инструментированием памяти и пытается поймать конкурентные обращения без синхронизации. Это очень полезный инструмент, но он не заменяет ревью.

У race detector есть важное ограничение: он находит только те гонки, которые реально произошли во время запуска тестов. Если тест не создал параллельную ситуацию, race detector ничего не покажет. Поэтому на ревью все равно нужно логически анализировать shared state.

На собеседовании можно сказать так:

> Я бы подтвердил это тестом с `go test -race`, но сам баг виден статически: одно и то же поле `cache` пишется без mutex и читается под mutex из другого метода.

Это хороший баланс: ты знаешь инструмент, но не перекладываешь мышление на инструмент.

## Какие тесты помогли бы

Для такого класса ошибок можно написать конкурентный тест, который параллельно вызывает `CreateOrder`, `GetOrder` и `RebuildCache`, а потом прогнать его с `-race`. Сам тест может не падать без `-race`, но race detector должен подсветить проблему.

Идея теста:

```go
func TestServiceCacheRace(t *testing.T) {
	s := newTestService()

	var wg sync.WaitGroup
	for i := 0; i < 100; i++ {
		wg.Add(2)

		go func(i int) {
			defer wg.Done()
			_, _ = s.GetOrder(context.Background(), strconv.Itoa(i))
		}(i)

		go func(i int) {
			defer wg.Done()
			_ = s.RebuildCache(context.Background(), "user-1")
		}(i)
	}

	wg.Wait()
}
```

Это не обязательно идеальный unit-тест, потому что нужны fake-репозитории, но как идея он показывает направление: создать параллельный доступ к одному shared state.

## Что говорить на собеседовании

Если на ревью-секции ты видишь подобный код, можно идти от общего к частному.

Сначала сказать:

```text
Я вижу shared mutable state: cache внутри Service. Service, судя по всему, долгоживущий и может использоваться параллельными запросами, значит доступ к cache должен быть синхронизирован.
```

Потом показать конкретику:

```text
В GetOrder cache читается и пишется под mutex, но в CreateOrder и RebuildCache запись идет напрямую. Это нарушает единый контракт доступа и дает data race.
```

Потом назвать последствия:

```text
Под нагрузкой это может дать concurrent map writes, неконсистентные данные или нестабильное поведение. Даже если локально не воспроизводится, это баг модели памяти.
```

Потом предложить решение:

```text
Я бы инкапсулировал cache за get/set/replace методами или вынес кеш в отдельный тип. Все обращения к map должны проходить через один mutex. Для RebuildCache лучше собрать данные локально и коротко заменить/обновить кеш под lock.
```

И затем можно добавить проверку:

```text
После фикса я бы добавил конкурентный тест и прогнал go test -race.
```

Такой ответ выглядит спокойно и профессионально. Ты показываешь, что понимаешь не только синтаксис mutex, но и жизненный цикл сервиса, модель владения данными, последствия и способ проверки.

## Короткая памятка для ревью

Когда видишь многопоточный Go-код, мысленно проходи по таким вопросам.

Есть ли здесь долгоживущий объект, который используется из разных горутин? Есть ли внутри него изменяемые поля? Есть ли `map`, `slice`, счетчики, кеши, каналы, worker'ы? Все ли обращения к этим данным идут через один и тот же механизм синхронизации? Не возвращаем ли мы наружу ссылку на внутреннее изменяемое состояние? Не держим ли mutex во время БД, сети или вызова чужого интерфейса? Есть ли понятный shutdown для горутин? Можно ли проверить это через `go test -race`?

Если пройти эти вопросы по коду, большинство race-проблем становится заметно даже без запуска программы.
