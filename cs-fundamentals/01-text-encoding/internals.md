# Кодировки текста: UTF-8 под капотом

Это тема-продолжение [B08: Строки в Go](../../base-go/08-strings/internals.md). Там разобрано, почему текст в компьютере вообще устроен как байты, откуда взялся Unicode и почему победил именно UTF-8 (раздел "От бита до буквы"), а также прикладные факты про Go-строку: `len` считает байты, `string` не гарантирует валидный UTF-8, `byte` и `rune` - разные слои.

Здесь - более глубокий, не привязанный конкретно к Go уровень: точная побитовая арифметика кодирования и декодирования UTF-8 вручную, какие последовательности байт запрещены и почему (overlong encoding, суррогатный диапазон), как устроена быстрая валидация под капотом, как именно декодирует `range` в Go, как ту же задачу решают другие языки и системы (Rust, Python, JavaScript, C, базы данных, файловые системы), и в чём разница между byte, code point (rune) и grapheme cluster.

Это общая computer-science база, а не Go-специфика - в тестах по теме "Строки в Go" эти вопросы сознательно не используются, чтобы не путать прикладную подготовку с подкапотной. Тема задумана как открытая для расширения: сюда же в будущем могут добавиться смежные темы вроде двоичного представления чисел, IEEE-754, endianness.

## Кодировка UTF-8: базовая схема

UTF-8 кодирует Unicode code point переменным числом байтов - от 1 до 4. Схема битов:

```text
code point диапазон      байты   битовый layout
U+0000  .. U+007F        1       0xxxxxxx
U+0080  .. U+07FF        2       110xxxxx 10xxxxxx
U+0800  .. U+FFFF        3       1110xxxx 10xxxxxx 10xxxxxx
U+10000 .. U+10FFFF      4       11110xxx 10xxxxxx 10xxxxxx 10xxxxxx
```

Почему выбрана именно такая схема (ASCII-совместимость, самосинхронизация, отсутствие конфликта с нулевым байтом) - подробно разобрано в [B08: раздел "От бита до буквы"](../../base-go/08-strings/internals.md). Здесь стоит добавить ещё одно полезное свойство, которое пригодится дальше: UTF-8 **сортируется побайтово** - лексикографическое сравнение байтов совпадает с сравнением code point, поэтому строки в Go можно сравнивать простым `<` по байтам, без декодирования.

Конкретные примеры (запомнить хотя бы один - сильно помогает на собесе):

```text
"A"      -> U+0041 -> 1 байт:  41
"é"      -> U+00E9 -> 2 байта: C3 A9
"П"      -> U+041F -> 2 байта: D0 9F   (кириллица вся 2-байтовая)
"世"     -> U+4E16 -> 3 байта: E4 B8 96 (CJK обычно 3-байтовые)
"👍"     -> U+1F44D-> 4 байта: F0 9F 91 8D (эмодзи вне BMP, 4 байта)
```

Отсюда классический пример из Go Blog:

```go
s := "Hello, 世界"
fmt.Println(len(s))                    // 13  (7 ASCII + 3 + 3)
fmt.Println(utf8.RuneCountInString(s)) // 9   (7 + 1 + 1)
```

7 ASCII-символов дают 7 байт, каждый иероглиф - 3 байта, итого 13 байт, но 9 рун.

И русский пример:

```go
s := "Привет"
fmt.Println(len(s))                    // 12  (6 букв * 2 байта)
fmt.Println(utf8.RuneCountInString(s)) // 6
```

До сих пор мы просто *смотрели* на готовые байты и верили табличке. Дальше - разбор того, откуда эти конкретные байты берутся: сначала совсем простыми словами, потом - с полной побитовой арифметикой, которую можно повторить руками на бумаге.

### Совсем просто: что вообще значит "закодировать один символ"

Возьмём самый простой случай - букву `A`. У неё есть Unicode-номер `U+0041`, то есть просто число 65. В двоичном виде 65 - это `1000001`, и это число помещается всего в 7 бит. Договорённость UTF-8 для такого маленького числа предельно проста: **используем ровно 1 байт, и этот байт - и есть само число**, без изменений. Байт `0x41` в памяти - это и есть буква `A`, буквально совпадает со старым ASCII.

Проблема начинается, когда номер символа больше, чем помещается в один байт. Например, у русской буквы `П` номер `U+041F` = 1055 - это уже больше 127, в один байт (максимум 255, но напомним: под ASCII занят диапазон 0-127, а Unicode не хочет путать эти два мира) такое число просто "в лоб" не запишешь однозначно. Нужна схема, которая говорит: "эти два байта на самом деле - один символ, а не два соседних маленьких числа". Именно для этого и придуманы префиксы `110` / `10`, `1110` / `10` / `10` и так далее - это метки-указатели: "я первый байт символа из N байт" или "я один из продолжающих байт".

Дальше - как именно эти метки и число символа складываются в конкретные байты, шаг за шагом.

### Пошаговое кодирование: от code point к байтам, вручную

Общий рецепт кодирования один и тот же для любого многобайтового случая:

1. Выбрать по таблице, сколько байт нужно (по диапазону, в который попадает code point).
2. Записать code point в двоичном виде.
3. Дополнить его слева нулями до точного количества "полезных" бит, которое умещается в выбранную схему (для 2 байт это `5 + 6 = 11` бит, для 3 байт - `4 + 6 + 6 = 16` бит, для 4 байт - `3 + 6 + 6 + 6 = 21` бит).
4. Разрезать эту битовую строку на куски нужной ширины - по одному куску на байт.
5. К первому куску приписать спереди метку ведущего байта (`110`, `1110` или `11110`), ко всем остальным - метку продолжения `10`.
6. Склеить получившиеся байты.

Разберём это вручную на `é` (`U+00E9`, десятичное 233). Диапазон `U+0080..U+07FF` -> нужно 2 байта, суммарно `5 + 6 = 11` полезных бит.

```text
233 в двоичном виде (8 бит):        11101001
дополняем нулями слева до 11 бит:   000 11101001
делим на куски 5 и 6 бит:           00011 | 101001

байт 1: 110 + 00011  = 11000011 = 0xC3
байт 2: 10  + 101001 = 10101001 = 0xA9

результат: C3 A9  <- ровно то, что было в таблице выше
```

Теперь то же самое для трёхбайтового случая - иероглифа `世` (`U+4E16`, десятичное 20 246). Диапазон `U+0800..U+FFFF` -> 3 байта, суммарно `4 + 6 + 6 = 16` полезных бит - как раз ровно 16 бит без всякого padding, потому что верхняя граница диапазона `U+FFFF` сама по себе 16-битная:

```text
0x4E16 в двоичном виде (16 бит):    0100 1110 0001 0110
делим на куски 4, 6 и 6 бит:        0100 | 111000 | 010110

байт 1: 1110 + 0100   = 11100100 = 0xE4
байт 2: 10   + 111000 = 10111000 = 0xB8
байт 3: 10   + 010110 = 10010110 = 0x96

результат: E4 B8 96  <- снова совпадает с таблицей
```

Четырёхбайтовый случай (эмодзи `👍`, `U+1F44D`) устроен идентично, только схема разбиения - `3 + 6 + 6 + 6 = 21` бит, и в результате получаются те же самые `F0 9F 91 8D` из таблицы выше. Механика везде одна и та же: посчитать биты, разложить по кусочкам заданной ширины, приклеить метки.

> На собесе такой ручной разбор - сильный сигнал: большинство кандидатов знают табличку "1-4 байта", но немногие могут показать, откуда конкретные байты берутся арифметически.

### Пошаговое декодирование: от байтов обратно к code point

Декодирование - это тот же рецепт в обратную сторону:

1. Посмотреть на первый байт. Посчитать, сколько единиц подряд стоит в начале, до первого нуля: ноль единиц (`0xxxxxxx`) - это самостоятельный ASCII-символ, одна ведущая группа `110` - значит дальше идёт ещё 1 байт-продолжение, `1110` - ещё 2, `11110` - ещё 3.
2. Отбросить у ведущего байта его метку, оставить только "полезные" биты.
3. У каждого продолжающего байта отбросить префикс `10`, оставить только 6 полезных бит.
4. Склеить оставшиеся биты по порядку в одну битовую строку.
5. Прочитать результат как обычное двоичное число - это и есть code point.

Проверим на тех же байтах `C3 A9`, которые мы только что получили для `é`:

```text
C3 = 11000011  -> метка 110 (2 байта), полезные биты: 00011
A9 = 10101001  -> метка 10 (продолжение), полезные биты: 101001

склеиваем: 00011 101001 = 00011101001 (11 бит)
как число: 233 = 0xE9 = U+00E9 = 'é'
```

Круг замкнулся: закодировали `é` в `C3 A9`, декодировали `C3 A9` обратно в `é`. Именно эту процедуру - определить длину по ведущему байту, снять маски, собрать биты обратно в число - выполняет `range` и функции пакета `unicode/utf8` под капотом (детали ниже).

### Overlong encoding: одна и та же буква - двумя разными наборами байт

Вот тонкость, которую редко объясняют, но которая - отличный senior-сигнал. В шаге кодирования мы дополняем число нулями слева до нужной ширины. А что, если применить схему "с запасом" - взять маленький code point вроде `A` (`U+0041`, 65) и всё равно закодировать его двухбайтовой схемой?

```text
65 в двоичном виде (7 бит):         1000001
дополняем нулями до 11 бит:         0000 1000001
делим на куски 5 и 6 бит:           00001 | 000001

байт 1: 110 + 00001  = 11000001 = 0xC1
байт 2: 10  + 000001 = 10000001 = 0x81

результат: C1 81
```

Формально при декодировании `C1 81` мы честно получим обратно число 65, то есть `A`. Но такая запись называется **overlong encoding** (избыточное, "не кратчайшее" кодирование), и стандарт UTF-8 её явно запрещает: один и тот же code point обязан кодироваться единственным, кратчайшим способом. Корректный декодер обязан **отклонить** `C1 81` как невалидную последовательность, даже если арифметика "сходится".

Почему это не формальная придирка, а реальная история про безопасность: в 1990-2000-х несколько известных систем сначала проверяли входную строку на запрещённые байты (например, символ `/` или последовательность `../` для запрета выхода из каталога), а уже потом декодировали её как UTF-8. Наивный или багованный декодер мог принять overlong-форму того же самого символа и раскодировать её в тот же опасный байт **уже после того, как проверка была пройдена** - то есть проверка на `../` не срабатывала, потому что во входных данных был не сам байт `.` или `/`, а его избыточно закодированный двух- или трёхбайтовый эквивалент. Это well-known класс уязвимостей "directory traversal через overlong UTF-8", и именно поэтому современный стандарт (RFC 3629) требует от декодера отбрасывать любую не-кратчайшую форму как невалидную, а не просто "тоже расшифровывать".

Отсюда практическое следствие для проверки валидности: байты `0xC0` и `0xC1` **не могут** легально встретиться как ведущий байт ни в одной корректной UTF-8-последовательности - потому что любая двухбайтовая последовательность, начинающаяся с `110 00000` или `110 00001`, может закодировать только code point из диапазона `0..127`, а такие code point обязаны кодироваться одним байтом. Поэтому в быстрых валидаторах `0xC0`/`0xC1` в роли лидирующего байта - это мгновенный, "бесплатный" признак битых данных.

В Go эту работу делает `utf8.ValidString`: она отклоняет overlong-формы, а `range`/`utf8.DecodeRuneInString` на такой последовательности вернут `utf8.RuneError` (`U+FFFD`) вместо "восстановленного" символа.

### Что ещё делает последовательность байт невалидной

Кроме overlong-кодирования, есть ещё несколько правил, которые стоит держать в голове:

- **Несовпадение количества продолжающих байт.** Ведущий байт объявляет: "дальше идёт ровно N байт-продолжений". Если их окажется меньше (строка неожиданно кончилась или встретился не `10xxxxxx` байт) или "осиротевший" байт-продолжение встретился без ведущего байта перед ним - последовательность невалидна.
- **Суррогатный диапазон `U+D800..U+DFFF` запрещён в UTF-8.** Эти номера зарезервированы исключительно для внутреннего механизма суррогатных пар UTF-16 (см. раздел про UTF-16 в [B08: Вторая проблема - как записать номер символа в байты](../../base-go/08-strings/internals.md)) и никогда не должны появляться как самостоятельно закодированные code point в UTF-8-потоке. Их присутствие - явный признак либо повреждения данных, либо ошибочной конвертации из UTF-16.
- **Максимум `U+10FFFF`.** Исторически ранняя спецификация UTF-8 (1993 год и её более поздняя стандартизация в середине 1990-х) допускала последовательности длиной до 6 байт и покрывала числа вплоть до `0x7FFFFFFF`. В 2003 году RFC 3629 сузил UTF-8 максимум до 4 байт и до `U+10FFFF` - то есть ровно до того предела, который вообще способен выразить UTF-16 через суррогатные пары. Это красивый исторический штрих: диапазон современного UTF-8 подогнан под ограничения конкурирующего формата, чтобы оба формата гарантированно покрывали одно и то же пространство символов.

### Как реально устроена быстрая валидация под капотом

Пошаговый алгоритм "посчитать единицы, отрезать биты, сложить число" отлично подходит для понимания, но производственные декодеры почти никогда не считают биты вручную в цикле - это было бы медленно и ветвисто (много `if`). Вместо этого широко известные высокопроизводительные реализации (например, decoder Бьёрна Хёрмана, "Flexible and Economical UTF-8 Decoder") устроены как маленький **table-driven конечный автомат**: для каждого входного байта из готовой таблицы одним обращением по индексу вычисляется сразу и "новое состояние" (сколько продолжающих байт ещё ожидается), и признак ошибки (в том числе overlong-формы и суррогаты отсекаются в том же самом проходе, без отдельной проверки). Идея в том, чтобы заменить дерево условных переходов на несколько табличных подстановок - это резко быстрее на современных процессорах, потому что почти нет непредсказуемых ветвлений.

Для понимания сути достаточно знать: концептуально это тот же самый алгоритм "посмотреть на биты ведущего байта -> определить длину -> проверить продолжающие байты -> собрать число", просто реализованный не через явные `if`/битовые маски в духе учебного примера выше, а через компактные lookup-таблицы ради скорости.

### Под капотом в Go: как декодируются range и utf8.DecodeRuneInString

Концептуально `utf8.DecodeRuneInString` делает ровно то же самое, что мы проделали руками, но начинает с быстрой проверки на самый частый случай:

```go
// Упрощённая идея, не точный исходный код:
func decodeRuneInString(s string) (r rune, size int) {
	if len(s) == 0 {
		return utf8.RuneError, 0
	}

	b0 := s[0]
	if b0 < 0x80 {
		// Быстрый путь: обычный ASCII-байт - это уже готовая руна.
		return rune(b0), 1
	}

	// Медленный путь: определить длину по ведущим битам b0,
	// проверить, что дальше идёт нужное число байт вида 10xxxxxx,
	// снять маски и собрать биты в itog rune.
	// Если что-то не сходится (мало байт, чужой префикс,
	// overlong-форма, суррогат, code point вне диапазона) -
	// вернуть utf8.RuneError и size=1.
}
```

Этот "быстрый путь" - не случайная мелкая оптимизация. Подавляющее большинство реального текста в бэкенд-коде - идентификаторы, JSON-ключи, заголовки протоколов, английские сообщения - это чистый ASCII, и для него достаточно одной проверки `b0 < 0x80`, без всякой битовой арифметики. Тяжёлый путь с масками и таблицами включается только тогда, когда байт действительно многобайтовый.

Ключевой концептуальный момент: **`range` по строке не содержит своей отдельной магии** - компилятор буквально вызывает эту же логику декодирования на каждом шаге цикла, получает пару `(rune, size)` и просто сдвигает индекс на `size`. Именно поэтому:

- индекс `range` "прыгает" не на 1, а на ширину только что декодированной руны;
- на невалидной последовательности (в том числе overlong или суррогатной) декодер возвращает `(utf8.RuneError, 1)`, и `range` продвигается всего на 1 байт, не падая и не зависая;
- поведение `range` полностью совпадает с тем, что делает `utf8.DecodeRuneInString`, потому что это буквально один и тот же код.

То есть всё поведение `range` (детали см. в [B08: range декодирует UTF-8](../../base-go/08-strings/internals.md)) - не отдельное свойство синтаксиса, а прямое следствие того, как устроен UTF-8-декодер, который мы только что построили руками.

### Как ту же задачу решают другие языки и системы

Полезно увидеть, что модель Go - не единственно возможная, и понимать, какой trade-off выбирают другие экосистемы для той же самой проблемы "байты vs символы".

**Rust** хранит строки (`String`, `&str`) тоже как UTF-8 байты - модель очень похожа на Go. Принципиальное отличие в другом: Rust делает валидность UTF-8 **гарантией на уровне типа**, а не просто конвенцией. Получить `String` из произвольных байт можно только через `String::from_utf8(bytes)`, которая возвращает `Result` и явно проваливается на невалидных данных, либо через сознательно небезопасную `from_utf8_unchecked` (помечена `unsafe`, ответственность на программисте). В Go же тип `string` в принципе ничего не гарантирует про валидность - это осознанный выбор в противоположную сторону того же самого спектра: Go экономит на проверках при каждой границе, Rust платит эту цену ради инварианта, который потом бесплатно держит компилятор.

**Python** (начиная с версии 3.3, PEP 393) устроен принципиально иначе: строка `str` внутри - это не UTF-8 байты, а массив code point с "гибкой" фиксированной шириной - 1 байт на символ, если все code point строки умещаются в Latin-1 (`<= 0xFF`), 2 байта, если умещаются в `0xFFFF`, иначе 4 байта. Это даёт O(1) произвольный доступ к N-му символу (в отличие от UTF-8, где до N-й руны надо декодировать по порядку), но требует явной кодировки в байты (`s.encode('utf-8')`) на любой границе с внешним миром - файлом, сетью, диском.

**JavaScript** исторически (с середины 1990-х, когда Unicode ещё помещался в 16 бит) выбрал внутреннее представление строки как последовательности **UTF-16 code unit**, а не code point. Отсюда знаменитая ловушка: `"😀".length === 2`, потому что эмодзи вне Basic Multilingual Plane кодируется суррогатной парой из двух 16-битных единиц, и наивная посимвольная итерация её ломает. Современный JS добавил `for...of` по строке, который корректно итерирует по code point, аналогично тому, как `range` в Go корректно итерирует по рунам поверх UTF-8 - разное внутреннее хранение, но одна и та же идея "дать безопасный способ обхода по смысловым единицам, а не по сырым единицам хранения".

**C/C++** не имеют вообще никакого встроенного понятия кодировки: `char*` - это просто байты, язык не знает и не заботится о том, что они означают. Вся Unicode-осведомлённость приходит только из внешних библиотек (ICU, iconv). Это в буквальном смысле тот исторический вакуум, что и породил хаос кодовых страниц, описанный в [B08](../../base-go/08-strings/internals.md), и который UTF-8 (и позже Plan 9/Go) были призваны исправить.

**Базы данных.** В PostgreSQL текст на диске обычно хранится как UTF-8 байты, но SQL-функция `LENGTH()` считает именно символы (декодирует UTF-8), а `OCTET_LENGTH()` - именно байты. Это ровно то же самое различие, что `len(s)` (байты) против `utf8.RuneCountInString(s)` (руны) в Go - разница байт/символ не Go-специфичная придумка, а фундаментальное свойство, всплывающее в любой системе, которая хранит текст как UTF-8.

**Файловые системы.** На Linux и большинстве POSIX-систем имя файла - это просто непрозрачная последовательность байт: файловая система **не требует**, чтобы оно было валидным UTF-8 (можно создать файл с "битым" именем, и многие утилиты будут с этим не совсем корректно работать). Windows NTFS, напротив, хранит имена файлов как последовательности UTF-16. Это прямая параллель со свойством Go-строки "может содержать невалидный UTF-8": это не языковая эксцентричность, а корректное отражение того, как реальные системы на самом деле обращаются с байтами на границе с внешним миром.

> На собесе сильная формулировка: "UTF-8 кодирует code point переменным числом байт, разбивая его двоичное представление на куски фиксированной ширины и помечая их префиксными битами; декодирование - это та же процедура в обратную сторону. Схема допускает только кратчайшую форму записи - overlong-кодирование запрещено явно, исторически из-за уязвимостей вроде обхода фильтров через избыточно закодированные `../`. Go реализует это как table-driven декодер с быстрым ASCII-путём, и `range` - это просто вызов этого декодера на каждом шаге. Другие языки решают тот же вопрос по-разному: Rust гарантирует валидность на уровне типа, Python хранит code point напрямую с адаптивной шириной, JavaScript унаследовал UTF-16 с суррогатными парами, а C вообще не знает о кодировках - Go сознательно выбрал 'просто байты плюс UTF-8 по конвенции' как самый дешёвый и универсальный вариант."

## rune, code point и grapheme - три разные вещи

Это тонкость senior-уровня, которую многие путают. Есть три уровня "единицы текста":

1. **byte** - физический байт UTF-8.
2. **rune / code point** - одно значение Unicode (`int32`). Результат декодирования UTF-8.
3. **grapheme cluster** - то, что человек воспринимает как один "символ" на экране.

Между ними нет равенства в общем случае:

```text
байты  >=  code points  >=  grapheme clusters
```

Примеры, где эти уровни расходятся:

- **Составные символы.** "é" можно записать одним code point `U+00E9` (precomposed) или двумя: `e` (`U+0065`) + combining acute `U+0301`. Второй вариант - 2 руны, но 1 grapheme, который выглядит одинаково.

```go
precomposed := "é"       // 1 руна, 2 байта
decomposed  := "é" // 2 руны, 3 байта, но визуально тот же "é"

fmt.Println(utf8.RuneCountInString(precomposed)) // 1
fmt.Println(utf8.RuneCountInString(decomposed))  // 2
fmt.Println(precomposed == decomposed)           // false! разные байты
```

- **Эмодзи с ZWJ.** Семейное эмодзи "👨‍👩‍👧" - это несколько эмодзи, склеенных zero-width joiner `U+200D`. Один grapheme на экране, но несколько code point и много байтов.
- **Флаги, модификаторы тона кожи, вариационные селекторы** - тоже один воспринимаемый символ из нескольких code point.

Практический вывод: вопрос "какой длины строка?" неоднозначен. Есть три разных ответа:

```go
s := "Привет👍"
len(s)                             // 16: байты (12 + 4)
utf8.RuneCountInString(s)          // 7:  code points (6 + 1)
// количество grapheme clusters    // 7 здесь, но в общем случае нужна
//                                 //   отдельная библиотека сегментации
```

В стандартной библиотеке нет разбиения на grapheme clusters - для этого нужны внешние пакеты (например, `rivo/uniseg`) или `golang.org/x/text`. На собеседовании достаточно чётко проговорить, что "символ" - размытое понятие, и уточнить, о байтах, рунах или grapheme идёт речь.

> На собесе: byte - это единица хранения, rune (code point) - единица Unicode, grapheme cluster - это то, что видит пользователь. `len` считает байты, `utf8.RuneCountInString` - руны, а количество grapheme в стандартной библиотеке не считается вообще. Комбинированные символы и ZWJ-эмодзи ломают наивное предположение "1 символ = 1 rune = N байт".
