# Идиомы Go: мапы, слайсы, сортировка

Это гайд-справочник по приёмам, а не тема для тестирования — сюда стоит
заглядывать, когда пишешь код и не помнишь, "как это принято делать в Go",
а не заучивать наизусть. Разобран конкретный кейс из [live-coding/01-top-k-frequent-words/top_k_frequent_words.go](../../live-coding/01-top-k-frequent-words/top_k_frequent_words.go)
(структура `Item` + `sort.Slice`), и вокруг него - более широкий набор идиом
для мап и слайсов с объяснением, что происходит под капотом и почему выбран
именно такой способ, а не наивный.

## Часть 1: Мапы

### Мапа как счётчик - без проверки на существование ключа

```go
counts := make(map[string]int)
for _, word := range words {
	counts[word]++
}
```

Это работает без предварительной проверки "а есть ли уже такой ключ",
потому что у чтения из мапы в Go есть встроенное поведение: **если ключа
нет, возвращается zero value типа значения**, а не паника и не ошибка. Для
`int` zero value - `0`, поэтому `counts[word]++` для нового слова
эквивалентно `counts[word] = 0; counts[word] = 0 + 1`, то есть `1`.

Наивный способ, который часто пишут, ещё не зная об этом:

```go
if _, ok := counts[word]; !ok {
	counts[word] = 0
}
counts[word]++
```

Работает так же, но лишний - авто-инициализация zero value уже даёт то же
самое поведение.

### comma-ok идиома: отличить "нет ключа" от "ключ есть, значение нулевое"

Проблема, которую решает zero-value-поведение выше: как отличить "слова
`"go"` в мапе вообще нет" от "слово `"go"` есть, но встретилось 0 раз"?
Обычное чтение `v := counts["go"]` не различает эти два случая - в обоих
`v` будет `0`. Для этого у чтения мапы есть вторая форма, возвращающая два
значения:

```go
v, ok := counts["go"]
// ok == true, если ключ реально есть в мапе
// v  == значение по ключу, либо zero value, если ключа нет
```

Это и есть comma-ok идиома - используется везде, где важно различить
"отсутствует" и "есть, но нулевое": в мапах, при чтении из канала
(`v, ok := <-ch`, чтобы отличить закрытый канал), при type assertion
(`v, ok := x.(int)`).

### Мапа как множество: `map[K]struct{}`

Если нужен просто набор уникальных значений без данных при них - идиома
Go не `map[K]bool`, а `map[K]struct{}`:

```go
seen := make(map[string]struct{})
for _, w := range words {
	seen[w] = struct{}{}
}
_, exists := seen["go"]
```

Почему `struct{}` вместо `bool`: пустая структура `struct{}` не занимает
памяти на инстанс (её размер - 0 байт), в отличие от `bool`, который
занимает байт на каждое значение. На множестве из миллиона строк разница
между `map[string]struct{}` и `map[string]bool` - это реальная экономия
памяти на значениях мапы (ключи всё равно хранятся в обоих случаях, разница
именно в значениях). `struct{}{}` выглядит непривычно при первой встрече,
но это устоявшийся идиоматичный способ сказать "меня интересует только сам
факт присутствия ключа".

### Почему обход мапы недетерминирован - и как это ловится

```go
for word, count := range counts {
	fmt.Println(word, count)
}
```

Порядок обхода мапы в Go **намеренно рандомизирован рантаймом** - это не
"не специфицировано и как получится", а сознательное инженерное решение:
разработчики Go на раннем этапе увидели, что если бы порядок был
детерминированным (например, по порядку вставки, как в некоторых других
языках), код начал бы незаметно на него полагаться, а такой код
ломался бы при малейших изменениях реализации хеш-таблицы. Рандомизация
обхода - защита от случайной зависимости от порядка там, где порядок
формально не гарантирован.

Практическое следствие: если нужен детерминированный порядок (например,
топ-K частых слов должен возвращаться в одном и том же порядке при каждом
запуске) - обход мапы напрямую не подходит. Нужно:

1. Собрать нужные данные из мапы в слайс.
2. Отсортировать слайс явно по нужному критерию.

Именно поэтому в `TopKFrequentWords` появляется промежуточный
`items := make([]Item, 0, len(counts))` - это не лишний шаг, а единственный
способ получить предсказуемый порядок из мапы.

### Предвыделение размера мапы: `make(map[K]V, n)`

```go
counts := make(map[string]int, expectedUniqueWords)
```

Как и у слайсов, у мапы есть внутренняя хеш-таблица, которая растёт по мере
добавления элементов и периодически перевыделяется под больший размер
(rehashing). Если заранее известна примерная итоговая величина - передача
её вторым аргументом `make` резервирует место сразу и избавляет от
нескольких промежуточных rehash по дороге. Не обязательно быть точным -
это подсказка планировщику памяти, а не жёсткий лимит (мапа продолжит расти
дальше, если элементов окажется больше).

## Часть 2: Слайсы

### Предвыделение capacity: `make([]T, 0, n)` против `var s []T`

```go
// Без подсказки: несколько реаллокаций по дороге, если элементов много
var items []Item
for k, v := range counts {
	items = append(items, Item{k, v})
}

// С подсказкой: одна аллокация на нужный размер сразу
items := make([]Item, 0, len(counts))
for k, v := range counts {
	items = append(items, Item{k, v})
}
```

`append` к слайсу без запаса `cap` в какой-то момент упирается в границу
capacity и выделяет новый backing array большего размера, копируя туда
старые элементы (подробный разбор роста capacity - в [B01: Слайсы](../../base-go/01-slices/review.md)).
Если конечный размер известен заранее (здесь - `len(counts)`, ровно
столько элементов и добавится), `make([]T, 0, n)` резервирует нужную
capacity одним выделением, и цикл `append` работает без единой
промежуточной реаллокации.

### Переиспользование буфера: `s[:0]` вместо нового слайса

Частый паттерн при накоплении временного буфера в цикле (например, при
сборке слова из рун посимвольно, как в `TopKFrequentWords`):

```go
var word []rune
for _, r := range text {
	if unicode.IsLetter(r) {
		word = append(word, r)
	} else {
		use(word)
		word = word[:0] // переиспользовать тот же backing array
	}
}
```

`word = word[:0]` обнуляет длину, но **сохраняет ту же выделенную память**
(`cap` не меняется) - следующий `append` пишет в уже готовый буфер без
новой аллокации, пока новое слово помещается в старую capacity. Сравни с
`word = nil` или `word = []rune{}` - оба варианта отвязывают переменную от
старой памяти, и следующий `append` начинает растить буфер с нуля заново,
то есть на каждое слово - минимум одна лишняя аллокация вместо
переиспользования. На тексте с тысячами слов разница ощутима: с `s[:0]`
буфер быстро "разгоняется" до capacity самого длинного встреченного слова и
дальше почти не аллоцирует, а без этого - аллокация на каждое слово заново.

### Фильтрация без лишней аллокации: `s = s[:0]` как аккумулятор

```go
nums := []int{1, 2, 3, 4, 5, 6}
evens := nums[:0]
for _, n := range nums {
	if n%2 == 0 {
		evens = append(evens, n)
	}
}
// evens == [2 4 6], тот же backing array, что и у nums
```

Это работает, потому что `append` в `evens` всегда пишет в позицию `<=`
текущей позиции чтения `n` в цикле `range nums` - "хвост" ещё не
прочитанных элементов исходного слайса никогда не перезаписывается раньше,
чем цикл до него дойдёт. Итог - фильтрация полностью на месте, без единой
новой аллокации, за счёт того, что результат заведомо не длиннее исходных
данных. Наивный вариант через `var evens []int` с `append` в отдельный
слайс тоже корректен, но платит за отдельный backing array, который здесь
не нужен.

## Часть 3: Сортировка - от `sort.Interface` до generics

Один и тот же результат (сортировка `[]Item` сначала по убыванию `Count`,
при равенстве - по возрастанию `Word`) можно получить тремя способами,
исторически появившимися в Go в разное время.

### Способ 1 - `sort.Interface` (классический, до дженериков)

```go
type byCountThenWord []Item

func (s byCountThenWord) Len() int      { return len(s) }
func (s byCountThenWord) Swap(i, j int) { s[i], s[j] = s[j], s[i] }
func (s byCountThenWord) Less(i, j int) bool {
	if s[i].Count != s[j].Count {
		return s[i].Count > s[j].Count
	}
	return s[i].Word < s[j].Word
}

sort.Sort(byCountThenWord(items))
```

Нужно завести именованный тип-обёртку над слайсом и реализовать три
метода интерфейса `sort.Interface` (`Len`, `Swap`, `Less`). Многословно, но
`sort.Sort` работает с этим типом напрямую, без reflection - `Swap` и
`Less` вызываются как обычные методы конкретного типа.

### Способ 2 - `sort.Slice` (Go 1.8, удобнее, но через reflection)

```go
sort.Slice(items, func(i, j int) bool {
	if items[i].Count != items[j].Count {
		return items[i].Count > items[j].Count
	}
	return items[i].Word < items[j].Word
})
```

Не нужен именованный тип - только функция сравнения по индексам. Удобнее в
использовании, но под капотом дороже: сигнатура `sort.Slice(x any, less
func(i, j int) bool)` принимает `x` как `any`, и сам факт индексного
доступа `s[i], s[j]` (swap) в общем виде для произвольного слайса
реализован через `reflectlite.Swapper(x)` - то есть каждый обмен местами
двух элементов идёт через reflection, а не через прямой доступ к памяти
типизированного слайса. Это прямо написано в комментарии к исходнику
`sort.Slice` в стандартной библиотеке Go: "in many situations, the newer
slices.SortFunc function is more ergonomic and runs faster".

### Способ 3 - `slices.SortFunc` (Go 1.21+, дженерики, рекомендуемый сейчас)

```go
slices.SortFunc(items, func(a, b Item) int {
	if a.Count != b.Count {
		return b.Count - a.Count // desc: сначала большее
	}
	return strings.Compare(a.Word, b.Word) // asc
})
```

Дженерик-версия: `SortFunc[S ~[]E, E any](x S, cmp func(a, b E) int)`.
Функция сравнения получает сами элементы (`a, b Item`), а не индексы, и
возвращает не `bool`, а знак разницы (`< 0`, `0`, `> 0`) - трёхстороннее
сравнение вместо "меньше/не меньше". Поскольку `E` - конкретный тип,
известный на этапе компиляции (через дженерики, а не `any`), swap
реализован без reflection - компилятор генерирует код под конкретный тип
`Item`, а не работает с произвольным `interface{}` в рантайме. Это и
причина, по которой `slices.SortFunc` в общем случае быстрее `sort.Slice`
на тех же данных - разница именно в способе доступа к элементам при
перестановке, а не в алгоритме сортировки (оба используют по сути один и
тот же pdqsort).

**Итог: на новом коде для Go 1.21+ предпочтителен `slices.SortFunc`** -
он быстрее `sort.Slice` и не требует многословности `sort.Interface`.
`sort.Slice` из `TopKFrequentWords` - не ошибка, а совершенно
жизнеспособный вариант (особенно в кодовой базе, где `slices` ещё не
принят как стандарт), но полезно знать, что есть более быстрая
современная альтернатива с тем же уровнем удобства.

### Стабильность сортировки: когда это важно

`sort.Slice` и `slices.SortFunc` **не гарантируют стабильность** - при
равенстве ключа сравнения относительный порядок "равных" элементов может
измениться после сортировки. Если это важно (например, сортируешь список
задач по приоритету, но хочешь, чтобы при равном приоритете сохранялся
исходный порядок добавления) - нужны `sort.SliceStable` или
`slices.SortStableFunc`, ценой немного более медленной сортировки
(гарантия стабильности не бесплатна). В `TopKFrequentWords` эта проблема
решена иначе - вторым ключом сравнения (`Word` по алфавиту), так что
"равных по всем ключам" элементов в принципе не возникает, и вопрос
стабильности не встаёт.

### Почему `Item{Word, Count}`, а не два параллельных слайса

Альтернативный, тоже рабочий способ хранить те же данные:

```go
words := make([]string, 0, len(counts))
freqs := make([]int, 0, len(counts))
for w, c := range counts {
	words = append(words, w)
	freqs = append(freqs, c)
}
```

Это называется **struct-of-arrays (SoA)** в противовес **array-of-structs
(AoS)**, которым и является `[]Item`. У SoA есть реальное преимущество в
узких высокопроизводительных сценариях: если в горячем цикле нужен доступ
только к одному полю (например, просуммировать все `freqs`, ни разу не
трогая `words`) - SoA даёт более плотное, cache-friendly расположение
именно нужных данных в памяти, без чтения "лишних" байт соседних полей.

Но здесь этот выигрыш не действует, а стоимость реальна: `words` и `freqs`
- это два независимых слайса, которые обязаны оставаться синхронизированы
по индексу вручную. Сортировка одного без параллельной сортировки другого
в лок-степе - классический источник багов "тихого рассинхрона данных",
который компилятор никак не поймает. `[]Item` с полями `Word` и `Count`
рядом в одной структуре сортируется одним вызовом и физически не может
рассинхронизироваться - слово и его счётчик всегда переставляются вместе,
потому что это один и тот же элемент слайса. Для кода такого масштаба
(не миллионы элементов в горячем цикле) простота и корректность
array-of-structs перевешивают гипотетический выигрыш SoA в кэше.

## Итоговая шпаргалка

- `m[key]++` работает без проверки существования - у мапы есть zero value.
- `v, ok := m[key]` - когда важно отличить "нет ключа" от "есть, но ноль".
- `map[K]struct{}` - идиоматичное множество, `struct{}` занимает 0 байт.
- Обход мапы недетерминирован специально - для стабильного порядка нужно
  извлечь в слайс и отсортировать явно.
- `make(map[K]V, n)` / `make([]T, 0, n)` - подсказка размера снижает число
  reallocation/rehash по дороге.
- `s = s[:0]` - переиспользовать buffer/фильтровать на месте без новой
  аллокации, вместо `s = nil` или `s = []T{}`.
- Сортировка: `sort.Interface` (без reflection, многословно) → `sort.Slice`
  (удобно, но через reflection) → `slices.SortFunc` (Go 1.21+, дженерики,
  быстрее `sort.Slice`, тот же уровень удобства) - для нового кода
  предпочтителен последний вариант.
- `sort.SliceStable`/`slices.SortStableFunc` - когда важен исходный
  порядок "равных" элементов.
- `[]Item{Word, Count}` вместо двух параллельных слайсов - array-of-structs
  проще и не может рассинхронизироваться; struct-of-arrays оправдан только
  в горячих циклах с доступом к одному полю на больших объёмах данных.
