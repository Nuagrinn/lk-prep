# Строки в Go: прикладное использование и ревью кода

Строки - тема, которая на собеседовании быстро отделяет "знаю синтаксис" от "понимаю модель". Почти все вопросы сводятся к одному: строка - это байты, а не символы. Обычно спрашивают:

- что такое строка в Go и immutable ли она;
- что вернёт `len` для строки с не-ASCII символами;
- что такое `rune` и чем он отличается от `byte`;
- что вернёт `s[i]`;
- как перебрать строку по символам;
- почему `range` по строке даёт не те индексы, что ждут;
- как развернуть строку с юникодом;
- почему конкатенация в цикле - это плохо;
- чем `string(65)` отличается от `"65"`;
- копирует ли `[]byte(s)`;
- безопасно ли шарить строку между горутинами.

Главная мысль:

> Строка в Go - это неизменяемая последовательность байтов, чаще всего в кодировке UTF-8. `len`, индексация и слайсинг работают в байтах; `range` и `[]rune` работают в рунах (code points). Символ, который видит человек, - это ещё третий уровень (grapheme), и он не равен ни байту, ни руне.

И вторая мысль:

> На ревью строки я проверяю не синтаксис, а границы: не путают ли байты с символами, не режут ли многобайтовую руну слайсингом, не строят ли строку через `+=` в цикле, не удерживают ли большой буфер подстрокой, и понимают ли, что строку не нужно защищать копией - она уже immutable.

## Простое введение

Строковые литералы бывают двух видов.

**Interpreted (в двойных кавычках)** - обрабатывают escape-последовательности:

```go
s := "line1\nline2\ttabé"
// \n перевод строки, \t табуляция, é -> é
```

**Raw (в обратных кавычках)** - берут текст как есть, без escape, можно многострочно:

```go
s := `C:\Users\name
no escapes here: \n остаётся \n`
```

Raw-строки удобны для регэкспов, путей Windows, SQL, JSON-шаблонов - там, где обратные слэши и кавычки иначе пришлось бы экранировать. Ограничение: внутри raw-строки нельзя поставить сам символ обратной кавычки.

Строка неизменяема:

```go
s := "hello"
s[0] = 'H' // compile error
```

Чтобы "изменить" строку, собирают новую (через `[]byte`, `[]rune` или `strings.Builder`).

## len - это байты, а не символы

Самая частая ошибка новичка - считать, что `len(s)` возвращает количество символов.

```go
fmt.Println(len("cat"))    // 3
fmt.Println(len("Привет")) // 12, а не 6
fmt.Println(len("世界"))    // 6, а не 2
fmt.Println(len("👍"))      // 4, а не 1
```

`len` - это длина в байтах, O(1). Чтобы посчитать руны:

```go
n := utf8.RuneCountInString("Привет") // 6
```

Не используй `len([]rune(s))` для подсчёта: это то же число, но с лишней аллокацией целого `[]rune`. `RuneCountInString` идёт по строке без выделения памяти.

На ревью:

> Если код проверяет длину пользовательского ввода (`len(name) > 20`) как "не больше 20 символов", а ввод может быть на кириллице/эмодзи - это баг: лимит окажется в 2-4 раза строже. Для лимита в символах нужен `utf8.RuneCountInString`.

## Индексация: байт против руны

`s[i]` - это `byte`, а не символ:

```go
s := "Hello"
fmt.Println(s[0])        // 72 (byte, код 'H')
fmt.Printf("%c\n", s[0]) // H
```

Для не-ASCII индексация по байту почти всегда бессмысленна:

```go
s := "Привет"
fmt.Printf("%c\n", s[0]) // Ð - половина руны 'П', мусор
```

Чтобы получить первый символ как руну:

```go
r, size := utf8.DecodeRuneInString(s) // r = 'П', size = 2
// или:
for _, r := range s {
	first := r
	break
}
```

Чтобы получить N-ю руну по индексу, придётся декодировать (`[]rune(s)[n]`), потому что байтовый доступ `s[n]` не соответствует символам.

## Обход строки: range против индекса

Два способа обхода дают принципиально разное (механику см. в [internals.md](internals.md)):

```go
s := "Привет"

// По байтам: 12 итераций, s[i] - байт
for i := 0; i < len(s); i++ {
	_ = s[i]
}

// По рунам: 6 итераций, r - руна, i - байтовое смещение
for i, r := range s {
	_ = i
	_ = r
}
```

Правило выбора:

- нужны **символы** (юникод-текст) - `range` или `[]rune`;
- нужны **байты** (бинарные данные, протоколы, заведомо ASCII) - индексный цикл `for i`;
- нужен **произвольный доступ по номеру символа** - `[]rune(s)` один раз, потом по индексу.

Частая ошибка - собрать индексы из `range`, думая, что они идут подряд:

```go
for i, r := range "aé" {
	fmt.Println(i, string(r))
}
// 0 a
// 1 é   <- следующий индекс был бы 3, а не 2
```

## Разворот строки

Наивный разворот по байтам ломает юникод:

```go
func reverseBytes(s string) string {
	b := []byte(s)
	for i, j := 0, len(b)-1; i < j; i, j = i+1, j-1 {
		b[i], b[j] = b[j], b[i]
	}
	return string(b) // для "Привет" получится битый UTF-8
}
```

Правильно разворачивать по рунам:

```go
func reverse(s string) string {
	r := []rune(s)
	for i, j := 0, len(r)-1; i < j; i, j = i+1, j-1 {
		r[i], r[j] = r[j], r[i]
	}
	return string(r)
}
```

Но даже это не идеально: разворот по рунам ломает grapheme clusters. Строка с комбинированными символами ("e" + combining accent) или ZWJ-эмодзи ("👨‍👩‍👧") развернётся некорректно, потому что руны переставятся, а связь между ними - нет. Абсолютно корректный разворот требует сегментации по grapheme (внешняя библиотека).

На собесе:

> Разворот строки по байтам ломает многобайтовые руны. По рунам (`[]rune`) - работает для большинства случаев, но ломает составные символы и эмодзи с ZWJ. Полностью корректный разворот требует разбиения на grapheme clusters, чего нет в стандартной библиотеке.

## Подстроки и удержание памяти

`s[i:j]` не копирует - подстрока делит backing с исходной строкой (детали в [internals.md](internals.md)). Отсюда две вещи для ревью.

**Слайсинг может разрезать руну:**

```go
s := "世界"
fmt.Println(utf8.ValidString(s[0:2])) // false: 2 байта из 3-байтовой руны
```

**Маленькая подстрока держит большой буфер:**

```go
func extractID(hugePayload string) string {
	return hugePayload[10:42] // держит весь hugePayload в памяти
}
```

Если результат живёт долго, а исходник большой, надо отвязаться копией:

```go
func extractID(hugePayload string) string {
	return strings.Clone(hugePayload[10:42]) // Go 1.20+
}
```

На ревью:

> Если функция возвращает маленькую подстроку большого буфера и результат долгоживущий (кладётся в кэш, структуру, глобал), стоит `strings.Clone`, иначе GC не освободит исходный буфер.

## Конверсии на практике

```go
b := []byte(s)  // копия, string -> []byte (mutable)
s := string(b)  // копия, []byte -> string
r := []rune(s)  // декод в code points, аллокация
s := string(r)  // энкод рун в UTF-8
```

Все копируют/аллоцируют (кроме известных no-alloc паттернов из [internals.md](internals.md)). Практические выводы:

- частые `[]byte(s)`/`string(b)` в горячем коде видны в профиле - минимизируй лишние конверсии;
- `[]rune(s)` бери только когда реально нужен индекс по символам или разворот, иначе `range`;
- изменение `[]byte(s)` **не** влияет на исходную строку - это копия:

```go
s := "hello"
b := []byte(s)
b[0] = 'H'
fmt.Println(s) // "hello" - строка не изменилась
```

### string(int) - ловушка

```go
n := 65
fmt.Println(string(n))            // "A" (code point U+0041), НЕ "65"
fmt.Println(strconv.Itoa(n))      // "65" - правильно
fmt.Println(fmt.Sprintf("%d", n)) // "65"
```

`go vet` предупреждает о `string(intVar)`. Осознанное "символ по коду" пишут явно: `string(rune(n))`.

## Построение строк: что выбрать

```go
// ПЛОХО: += в цикле, O(n^2) по копированию
var s string
for _, x := range items {
	s += x
}

// ХОРОШО: strings.Builder, O(n)
var b strings.Builder
b.Grow(estimate) // опционально
for _, x := range items {
	b.WriteString(x)
}
s := b.String()
```

Шпаргалка по выбору:

| Ситуация | Инструмент |
|---|---|
| Склеить срез с разделителем | `strings.Join(items, ", ")` |
| Собрать строку в цикле | `strings.Builder` |
| Нужен `io.Writer` попутно | `bytes.Buffer` |
| Разовое `a + b + c` | обычный `+` (компилятор оптимизирует) |
| Форматирование с типами | `fmt.Sprintf` (удобно, но медленнее) |
| Повторить строку N раз | `strings.Repeat(s, n)` |

`strings.Builder` отдаёт результат без копии; `bytes.Buffer.String()` копирует. Если строка только собирается и возвращается - `Builder` эффективнее.

На ревью:

> `+=` в цикле - красный флаг: O(n^2). Замена на `strings.Builder` или `strings.Join`. Если размер известен - `Builder.Grow` убирает промежуточные реаллокации.

## Пакет strings: частые операции

```go
strings.Contains(s, "ab")        // есть ли подстрока
strings.HasPrefix(s, "http://")  // начинается с
strings.HasSuffix(s, ".json")    // заканчивается на
strings.Index(s, "x")            // байтовый индекс первого вхождения или -1
strings.Count(s, "a")            // число непересекающихся вхождений
strings.Split(s, ",")            // []string по разделителю
strings.SplitN(s, ",", 2)        // максимум N частей
strings.Fields(s)                // разбить по любым пробелам, без пустых
strings.Join(parts, ", ")        // склеить
strings.Replace(s, "a", "b", -1) // заменить (n=-1: все)
strings.ReplaceAll(s, "a", "b")  // то же, читабельнее
strings.TrimSpace(s)             // убрать пробелы по краям
strings.TrimPrefix(s, "v")       // убрать префикс, если есть
strings.Trim(s, "/")             // убрать заданные символы по краям
strings.Repeat("ab", 3)          // "ababab"
```

`strings.Cut` (Go 1.18) - идиоматичный способ разбить "ключ=значение" на две части:

```go
key, val, found := strings.Cut("name=vlad", "=")
// key="name", val="vlad", found=true
```

Он чище, чем `SplitN(s, "=", 2)` с проверкой длины.

### Регистр и сравнение без учёта регистра

```go
strings.ToUpper("go")  // "GO"
strings.ToLower("GO")  // "go"
```

`ToUpper`/`ToLower` - Unicode-aware, но **не** локале-зависимы. Классический подвох - турецкая `i`: в турецкой локали заглавная от `i` - это `İ` (с точкой), а строчная от `I` - `ı` (без точки). Go всегда использует дефолтные Unicode-маппинги, поэтому `ToLower("I") == "i"` независимо от локали. Для локале-корректного регистра нужен `golang.org/x/text/cases`.

Для сравнения без учёта регистра **не** пиши `ToLower(a) == ToLower(b)` - это две лишние аллокации и не всегда корректно. Используй:

```go
strings.EqualFold("Go", "GO") // true, Unicode case folding, без аллокаций
```

`strings.Title` deprecated (Go 1.18) из-за неправильной обработки словных границ - замена в `golang.org/x/text/cases`.

## Пакеты unicode и unicode/utf8

```go
utf8.RuneCountInString(s) // число рун
utf8.ValidString(s)       // валидный ли UTF-8
utf8.DecodeRuneInString(s)// первая руна и её ширина
utf8.RuneLen('世')         // сколько байт занимает руна (3)

unicode.IsLetter(r) // буква
unicode.IsDigit(r)  // цифра
unicode.IsSpace(r)  // пробельный
unicode.ToUpper(r)  // руна в верхний регистр
```

Проверка ввода на валидный UTF-8 (важно для данных из сети/файлов):

```go
if !utf8.ValidString(input) {
	return errors.New("invalid UTF-8")
}
```

## Иммутабельность в дизайне API

Здесь строки радикально отличаются от слайсов и map, и это сильный пункт на собесе.

Слайс/map, возвращённые наружу, нужно защищать копией - иначе caller мутирует внутреннее состояние:

```go
func (s *Store) Items() []Item {
	return append([]Item(nil), s.items...) // защитная копия обязательна
}
```

Строку возвращать безопасно **без копии** - её нельзя изменить:

```go
func (u *User) Name() string {
	return u.name // никакой защитной копии не нужно
}
```

Точно так же строку-параметр можно спокойно сохранять в структуре: caller не сможет её потом поменять. Это делает строки идеальным типом для идентификаторов, ключей, неизменяемых конфигов.

На ревью:

> Возврат внутреннего слайса/мапы наружу - потенциальный баг (нужна копия). Возврат строки - безопасен, потому что строка immutable. Не нужно "защищать" строку `strings.Clone`, кроме случая отвязки от большого backing-буфера.

## Строки и конкурентность

Строку безопасно читать и шарить из любого числа горутин без синхронизации - она неизменяема, записи нет, значит нет data race:

```go
var config = "prod" // безопасно читать из многих горутин
```

Контраст с map и слайсом, где конкурентная запись требует mutex. Единственная оговорка - если строку переприсваивают (`config = "..."` из одной горутины, читают из другой), то гонка идёт по переменной, а не по байтам; для этого нужна синхронизация (mutex/atomic.Value), но сами байты строки при этом целостны.

## Строки и JSON

```go
type Msg struct {
	Text string `json:"text"`
	Data []byte `json:"data"`
}
```

Нюанс для ревью: поле `[]byte` в `encoding/json` кодируется как **base64-строка**, а не как массив чисел:

```json
{"text":"hi","data":"aGVsbG8="}
```

Это часто удивляет. Если нужен массив байт как числа - нужен кастомный маршалинг. `string` кодируется как обычная JSON-строка с экранированием.

## Частые задачи "что выведет"

### Задача 1: len юникода

```go
s := "Hello, 世界"
fmt.Println(len(s))
fmt.Println(utf8.RuneCountInString(s))
```

```text
13
9
```

### Задача 2: индексация даёт байт

```go
s := "Привет"
fmt.Println(s[0])
fmt.Printf("%c\n", s[0])
```

```text
208
Ð
```

`208` = `0xD0`, первый байт двухбайтовой руны 'П'.

### Задача 3: индексы range

```go
for i, r := range "aé" {
	fmt.Println(i, string(r))
}
```

```text
0 a
1 é
```

Второй индекс - `1`, не `2`. Следующий байтовый индекс был бы `3`.

### Задача 4: string(int)

```go
fmt.Println(string(65))
fmt.Println(strconv.Itoa(65))
```

```text
A
65
```

### Задача 5: слайсинг режет руну

```go
s := "世界"
fmt.Println(len(s))
fmt.Println(utf8.ValidString(s[0:2]))
```

```text
6
false
```

### Задача 6: []byte - это копия

```go
s := "hello"
b := []byte(s)
b[0] = 'H'
fmt.Println(s, string(b))
```

```text
hello Hello
```

Строка не изменилась: конверсия скопировала байты.

### Задача 7: подсчёт "символов" индексным циклом

```go
s := "Привет"
count := 0
for i := 0; i < len(s); i++ {
	count++
}
fmt.Println(count)
```

```text
12
```

Индексный цикл считает байты. Для рун нужен `range` или `RuneCountInString`.

### Задача 8: precomposed vs decomposed

```go
a := "é"        // U+00E9
b := "é"  // e + combining acute
fmt.Println(a == b)
fmt.Println(len(a), len(b))
```

```text
false
2 3
```

Выглядят одинаково, но разные байты. Для равенства нужна Unicode-нормализация.

## Типовые ошибки на ревью

### Ошибка 1. len как количество символов

```go
if len(username) > 16 {
	return errTooLong
}
```

Для юникода лимит окажется в разы строже. Нужно `utf8.RuneCountInString`, если лимит в символах.

### Ошибка 2. Индексация не-ASCII по байту

```go
firstLetter := s[0] // байт, а не символ
```

Для юникода нужно `utf8.DecodeRuneInString` или `range`.

### Ошибка 3. += в цикле

```go
var out string
for _, line := range lines {
	out += line + "\n"
}
```

O(n^2). Заменить на `strings.Builder`.

### Ошибка 4. string(int) вместо strconv

```go
id := "user-" + string(userNum) // мусорный символ
```

Нужно `strconv.Itoa` или `fmt.Sprintf`.

### Ошибка 5. Слайсинг многобайтовой строки по "символам"

```go
preview := text[:100] // может разрезать руну
```

Если `text` юникодный, `[:100]` режет по байтам и может дать битый UTF-8. Нужно резать по рунам или проверять границу.

### Ошибка 6. ToLower для сравнения без регистра

```go
if strings.ToLower(a) == strings.ToLower(b) {
```

Две аллокации и не всегда корректно. Нужно `strings.EqualFold(a, b)`.

### Ошибка 7. Удержание большого буфера подстрокой

```go
cache[key] = bigResponse[start:end]
```

Держит весь `bigResponse`. Нужно `strings.Clone`.

### Ошибка 8. Разворот/обработка по байтам

```go
b := []byte(s)
// переставляем байты
```

Ломает многобайтовые руны. Работать по `[]rune`, помня про grapheme.

### Ошибка 9. Неоправданный unsafe для zero-copy

```go
s := unsafe.String(unsafe.SliceData(b), len(b))
```

Если `b` потом мутируется/переиспользуется - строка "поедет". Опасно без чёткого owned-контракта; в 99% случаев обычная конверсия правильнее (а в map-key/сравнении компилятор и так не аллоцирует).

### Ошибка 10. Защитная копия строки "на всякий случай"

```go
return strings.Clone(u.name)
```

Строка immutable, лишняя копия не нужна (кроме отвязки от большого backing). Это анти-паттерн, обратный ошибке 7.

## Практики использования

### Подсчёт рун

```go
n := utf8.RuneCountInString(s)
```

### Первый символ как руна

```go
r, _ := utf8.DecodeRuneInString(s)
```

### Безопасная подстрока по рунам

```go
func runeSubstr(s string, from, to int) string {
	r := []rune(s)
	if from < 0 { from = 0 }
	if to > len(r) { to = len(r) }
	return string(r[from:to])
}
```

### Построение строки

```go
var b strings.Builder
b.Grow(len(a) + len(sep)*n)
for i, x := range items {
	if i > 0 {
		b.WriteString(sep)
	}
	b.WriteString(x)
}
result := b.String()
```

### Сравнение без учёта регистра

```go
if strings.EqualFold(a, b) {
	// ...
}
```

### Разбор "ключ=значение"

```go
key, val, ok := strings.Cut(line, "=")
if !ok {
	return errBadLine
}
```

### Валидация UTF-8 на входе

```go
if !utf8.ValidString(input) {
	return errInvalidUTF8
}
```

### Отвязка подстроки от большого буфера

```go
id := strings.Clone(payload[start:end])
```

## Как отвечать на собеседовании

### Что такое строка в Go?

> Строка - это неизменяемый header из указателя на байты и длины в байтах. Байты обычно в UTF-8, но тип этого не гарантирует. Строка не хранит символы или кодировку - только байты, а интерпретация как текста это соглашение.

### Immutable ли строка и зачем?

> Да. `s[i] = x` не компилируется. Иммутабельность даёт дешёвое разделение байтов (подстрока не копирует), безопасное чтение из многих горутин, стабильные ключи map и хранение литералов в read-only памяти.

### Что вернёт len для "世界"?

> 6. `len` считает байты, а каждый иероглиф в UTF-8 занимает 3 байта. Рун здесь 2, их считает `utf8.RuneCountInString`.

### byte vs rune?

> `byte` это `uint8`, единица хранения. `rune` это `int32`, один Unicode code point, результат декодирования UTF-8. Строка состоит из байтов; руны получаются декодированием.

### Что вернёт s[i]?

> Байт (`byte`), O(1). Для ASCII это совпадает с символом, для многобайтовых рун - лишь фрагмент, обычно бессмысленный.

### Как перебрать строку по символам?

> `for i, r := range s`: `r` это руна, `i` это байтовое смещение начала руны, шаг равен ширине руны. Или `[]rune(s)` для произвольного доступа по индексу.

### Почему range даёт "неправильные" индексы?

> Потому что индекс - это байтовое смещение, а не порядковый номер символа. После многобайтовой руны индекс перепрыгивает на её ширину.

### Как развернуть строку с юникодом?

> По рунам через `[]rune`. Побайтовый разворот ломает многобайтовые руны. Полностью корректно - по grapheme clusters, но этого нет в стандартной библиотеке.

### Почему += в цикле плохо?

> Каждая конкатенация создаёт новую строку и копирует всё накопленное, потому что строки immutable. В цикле это O(n^2). Нужен `strings.Builder` или `strings.Join`.

### Копирует ли []byte(s)?

> Да, обычно копирует, потому что `[]byte` mutable, а `string` immutable. Исключения - паттерны, где компилятор доказывает безопасность и убирает аллокацию (ключ map, сравнение с литералом, range).

### string(65) это "65"?

> Нет, это "A": число трактуется как code point. Для числа в строку нужен `strconv.Itoa`. `go vet` предупреждает о `string(int)`.

### Безопасно ли шарить строку между горутинами?

> Да, читать - безопасно без синхронизации, потому что строка неизменяема. Синхронизация нужна только если переменную со строкой переприсваивают.

## Методология ревью кода со строками

1. Проверить `len`: не используется ли как количество символов для юникод-данных.
2. Проверить индексацию: `s[i]` для не-ASCII - это байт, не символ.
3. Проверить обход: нужны символы - `range`/`[]rune`, нужны байты - индексный цикл.
4. Проверить слайсинг: не режет ли многобайтовую руну; не удерживает ли большой буфер.
5. Проверить построение: нет ли `+=` в цикле; уместен ли `Builder`/`Join`.
6. Проверить конверсии: нет ли лишних `[]byte`/`[]rune` в горячем коде.
7. Проверить `string(int)`: нет ли случайной конверсии числа в code point.
8. Проверить сравнение регистра: `EqualFold` вместо `ToLower == ToLower`.
9. Проверить регистр: понимают ли, что `ToUpper/ToLower` не локале-зависимы.
10. Проверить валидацию: данные из сети/файлов - `utf8.ValidString`.
11. Проверить память: долгоживущая подстрока большого буфера - `strings.Clone`.
12. Проверить immutable-дизайн: нет ли лишних защитных копий строк и нет ли, наоборот, незащищённых слайсов/мап.
13. Проверить `unsafe`-конверсии: есть ли реальный owned-контракт или это преждевременная оптимизация.

## Мини-чеклист

- `len` не путается с количеством символов?
- Индексация по байту не применяется к юникоду как к символам?
- Обход по рунам там, где нужны символы?
- Слайсинг не режет многобайтовую руну?
- Долгоживущая подстрока не держит большой буфер (`strings.Clone`)?
- Нет `+=` в цикле (есть `Builder`/`Join`)?
- `Builder.Grow`, если размер известен?
- Нет `string(int)` вместо `strconv`?
- Сравнение без регистра через `EqualFold`?
- Учтено, что `ToUpper/ToLower` не локале-зависимы?
- Вход из сети валидируется на UTF-8?
- Нет лишних конверсий `[]byte`/`[]rune` в горячем пути?
- Строки не защищаются копией без нужды (они immutable)?
- `unsafe`-конверсии имеют чёткий контракт неизменяемости?

## Короткое резюме

Строка в Go - это неизменяемые байты, а не символы:

- `len`, `s[i]`, `s[i:j]` работают в байтах;
- `range`, `[]rune` работают в рунах;
- символ на экране (grapheme) - третий уровень, не равный ни байту, ни руне;
- строка immutable: её безопасно шарить и возвращать без копий;
- конверсии `[]byte`/`[]rune` копируют;
- `string(int)` это code point, не число;
- `+=` в цикле - O(n^2), нужен `Builder`;
- слайсинг делит backing: не режь руну и не держи большой буфер;
- сравнение побайтовое, не локале-зависимое; для регистра - `EqualFold`.

Сильная формулировка для собеса:

> Я держу в голове три уровня: байт (хранение), rune (Unicode code point) и grapheme (то, что видит пользователь). Большинство багов со строками - это спутанные уровни: `len` в символах, индексация руны по байту, слайсинг посреди руны или `+=` в цикле. А ещё строка immutable, поэтому в отличие от слайса и map её не надо защищать копией - кроме случая, когда маленькая подстрока держит большой буфер.
